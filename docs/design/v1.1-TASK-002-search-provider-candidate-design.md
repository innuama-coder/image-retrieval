This document defines the v1.1 detailed design for image search providers,
provider readiness, weighted scheduling, search request/response DTOs,
candidate normalization, dedupe, provenance, provider diagnostics, and the
handoff contract into candidate quality and retrieval planning.

This is documentation-only detailed design. It does not implement code, tests,
or Cargo manifest changes.

## Scope And Boundary

### In Scope

- Canonical `BaseSearchProvider` trait and compatibility strategy for the
  current `BaseProvider` trait.
- Provider registry, adapter lookup, readiness evaluation, and config mapping
  from TASK-001 `RuntimeConfig.providers`.
- `SearchRequest`, `SearchResponse`, raw result DTO, normalized candidate DTO,
  provider usage events, diagnostics, and shortage reasons.
- Weighted random scheduling algorithm for multiple ready providers.
- Candidate target calculation consumption:
  `NormalizedQueryPlan.candidate_target = required_image_count * 20`.
- Candidate normalization fields: source URL, image URL, thumbnail URL,
  provider rank, search round, license hint, dimensions, MIME type, provenance,
  dedupe key, and origin candidate IDs.
- Dedupe policy across provider outputs and repeated search rounds.
- Fixture-vs-production boundaries for search providers and scheduler tests.
- Redaction, security, observability, metrics, and package handoff.

### Forbidden Scope And Non-Goals

- Do not implement code in this task.
- Do not design retrieval execution outside candidate data needed by retrieval.
- Do not design mechanical candidate quality scoring or Qwen 3.5 VLM subjective
  evaluation. TASK-003 owns that.
- Do not implement provider adapters in this task. The product decision for
  the default broad provider is now SerpApi Google Images; this document only
  defines the adapter/config contract.
- Do not hard-code provider credentials, API keys, endpoints, or paid/provider
  policy into production source.
- Do not let fixture providers count as production search evidence.

## Current Source Delta

The current Rust code has a useful MVP baseline:

- `src/ports/mod.rs` exposes `BaseProvider` with `provider_id`,
  `display_name`, `readiness`, `weight`, and `search(query, max_results)`.
- `src/search/registry.rs` stores `ProviderRegistration` and builds a weight
  table with readiness overrides.
- `src/search/scheduler.rs` already has a pluggable `RandomSource`, weighted
  selection, invocation cap, dedupe by `source_url`, usage events, and shortage
  reasons.
- `src/search/fixture.rs` provides test-only fixture providers.
- `src/domain/candidate.rs` has minimal candidate identifiers and dimensions.
- `src/domain/search.rs` has first-pass readiness, request, usage, outcome, and
  shortage types.

The v1.1 gap is that these shapes are still too narrow for production search:

- The canonical trait name must be `BaseSearchProvider` or an explicit alias.
- `readiness()` must return structured evidence, not only `Result<()>`.
- `search()` must accept a structured `SearchRequest`, not a string and count.
- Search responses must preserve provider-specific rank, raw result identity,
  search round, source page URL, image URL, license hints, dimensions, MIME
  type, and normalization diagnostics.
- Dedupe must preserve `origin_candidate_ids` and provenance instead of simply
  dropping duplicate URLs.
- Scheduler output must be auditable by delivery package files, especially
  `image-recalls.json` and `retrieval-manifest.json`.

## Interfaces Types And DTO Contracts

### Canonical Search Provider Trait

`BaseSearchProvider` is the v1.1 canonical provider contract. During migration,
`BaseProvider` may remain as a compatibility alias:

```rust
pub trait BaseSearchProvider: Send + Sync {
    fn provider_id(&self) -> ProviderId;
    fn display_name(&self) -> &str;
    fn provider_kind(&self) -> SearchProviderKind;
    fn supported_constraints(&self) -> ProviderConstraintSupport;
    fn readiness(&self, config: &SearchProviderConfig) -> ProviderReadinessReport;
    fn search(&self, request: &SearchRequest) -> Result<SearchResponse, SearchError>;
}

pub trait BaseProvider: BaseSearchProvider {}
```

Migration rule:

- New production adapters implement `BaseSearchProvider`.
- Existing fixture/stub providers may implement `BaseSearchProvider` directly.
- `BaseProvider` remains only if needed for existing tests and should delegate
  to the canonical contract.
- `BaseSearchProvider::readiness` must not perform a full search. It may check
  config shape, credential env var presence, endpoint parseability, health
  endpoint, quota signal, and provider retirement status according to adapter
  capability.

### Provider Config Input

TASK-001 defines `SearchProviderConfig`. TASK-002 consumes it with the following
provider-specific constraints:

```rust
pub struct SearchProviderConfig {
    pub provider_id: String,
    pub provider_kind: SearchProviderKind,
    pub enabled: bool,
    pub weight: u32,
    pub endpoint: Option<String>,
    pub credential_env: Option<String>,
    pub default_query_params: BTreeMap<String, String>,
}
```

Rules:

- `provider_id` is stable and appears in package evidence.
- `provider_kind` is the adapter family: `serpapi_google_images`, `commons`,
  `custom_http`, or `fixture`. Additional provider kinds such as `brave`,
  `serper`, or `google_cse` may be added later behind the same trait, but they
  are not the v1.1 default.
- `enabled = false` excludes the provider from scheduling but still appears in
  readiness reports.
- `weight = 0` for an enabled provider is `PROVIDER_WEIGHT_INVALID`.
- `credential_env` names an environment variable. The resolved value is private
  to the adapter and never appears in DTOs, logs, diagnostics, or package files.
- `fixture` kind is allowed only in test or explicit fixture mode. It is
  `PROVIDER_FIXTURE_NOT_PRODUCTION` in production runs.

### Default SerpApi Google Images Provider

The v1.1 default broad image search provider is SerpApi Google Images.
Development must provide a production adapter with these config defaults unless
the runtime config explicitly overrides them:

```toml
[[providers]]
provider_id = "serpapi_google_images"
provider_kind = "serpapi_google_images"
enabled = true
weight = 100
endpoint = "https://serpapi.com/search"
credential_env = "SERPAPI_API_KEY"

[providers.default_query_params]
engine = "google_images"
```

Adapter rules:

- The adapter sends `engine=google_images`, `q`, and the resolved
  `SERPAPI_API_KEY` value as private request parameters. The API key value must
  never enter `SearchRequest`, diagnostics, logs, or package files.
- Optional locale or filtering parameters may be passed only from
  `default_query_params` or future explicit QueryPlan/provider policy fields.
  They must not be hard-coded in source.
- The adapter maps SerpApi `image_results[]` entries into
  `ProviderRawImageResult` and then into canonical `CandidateRecord` values.
- Candidate `provider_rank` is the one-based index in `image_results` unless
  SerpApi provides a stronger rank field.
- SerpApi result fields that correspond to original image URL, source page URL,
  thumbnail URL, title, source, dimensions, or MIME/type evidence should be
  mapped when present. Missing optional fields produce diagnostics rather than
  fabricated values.
- Readiness requires the provider config to be enabled, positive-weight,
  non-fixture, endpoint-parseable, and to name `SERPAPI_API_KEY` or an explicit
  overriding credential env var that is present at runtime.
- Real-service smoke for search must prove at least one normalized candidate
  came from this adapter when SerpApi is configured as the only broad provider.

### Provider Registry

The registry owns provider registration metadata and adapter references. It
does not own raw credentials.

```rust
pub struct ProviderRegistry {
    pub registrations: BTreeMap<ProviderId, ProviderRegistration>,
    pub adapters: BTreeMap<ProviderId, Arc<dyn BaseSearchProvider>>,
}

pub struct ProviderRegistration {
    pub provider_id: ProviderId,
    pub display_name: String,
    pub provider_kind: SearchProviderKind,
    pub enabled: bool,
    pub configured_weight: u32,
    pub config_fingerprint: String,
    pub fixture_only: bool,
    pub capabilities: ProviderConstraintSupport,
}
```

Registry construction:

1. Load `RuntimeConfig.providers`.
2. For each config record, locate an adapter by `provider_kind`.
3. If no adapter exists, create a readiness report with
   `PROVIDER_ADAPTER_MISSING`.
4. If adapter exists, create `ProviderRegistration` using package-safe config
   values only.
5. Call adapter readiness and merge config and adapter evidence.
6. Build an effective weight table only from providers that are enabled, ready,
   positive-weight, non-fixture in production, and constraint-compatible.

### Provider Readiness Report

The MVP `ProviderReadiness` enum should be upgraded into a report while keeping
an enum summary for self-check display.

```rust
pub enum ProviderReadinessStatus {
    Ready,
    Disabled,
    MissingCredentials,
    Misconfigured,
    HealthFailed,
    QuotaExhausted,
    ConstraintUnsupported,
    Retired,
    FixtureOnly,
    Unavailable,
}

pub struct ProviderReadinessReport {
    pub provider_id: ProviderId,
    pub provider_kind: SearchProviderKind,
    pub display_name: String,
    pub status: ProviderReadinessStatus,
    pub available: bool,
    pub included_in_weight_table: bool,
    pub configured_weight: u32,
    pub effective_weight: Option<u32>,
    pub credential_status: CredentialStatus,
    pub health_check_status: HealthCheckStatus,
    pub quota_status: QuotaStatus,
    pub constraint_support: ProviderConstraintSupport,
    pub failure_code: Option<ProviderFailureCode>,
    pub checked_at: OffsetDateTime,
    pub evidence: Vec<ProviderEvidence>,
    pub redaction_applied: bool,
}
```

Required failure codes:

| Code | Meaning |
| --- | --- |
| `PROVIDER_DISABLED` | Config explicitly disables provider. |
| `PROVIDER_ADAPTER_MISSING` | Config names a provider kind not compiled or registered. |
| `PROVIDER_CREDENTIAL_MISSING` | Required credential env var is absent. |
| `PROVIDER_WEIGHT_INVALID` | Enabled provider has zero or invalid weight. |
| `PROVIDER_HEALTH_FAILED` | Adapter health check failed. |
| `PROVIDER_QUOTA_EXHAUSTED` | Provider indicates quota/rate limit exhaustion. |
| `PROVIDER_CONSTRAINT_UNSUPPORTED` | Provider cannot satisfy QueryPlan/provider policy constraints. |
| `PROVIDER_RETIRED` | Provider is known unavailable, deprecated, or blocked by policy. |
| `PROVIDER_FIXTURE_NOT_PRODUCTION` | Fixture adapter was selected in production mode. |
| `PROVIDER_UNAVAILABLE` | Unknown or transient provider unavailability. |

Readiness is a run gate:

- If zero providers are available, search returns `SearchSessionOutcome` with
  `NO_AVAILABLE_SEARCH_PROVIDER` and the orchestrator blocks the attempt before
  candidate quality or retrieval planning.
- If SerpApi is configured as the default broad provider but its adapter,
  credential env var, endpoint, or quota is unavailable, self-check reports the
  corresponding provider readiness failure. A run may proceed only if another
  configured real provider is ready and policy allows fallback.

### Search Request

`SearchRequest` is passed to providers and is package-safe. It includes the
normalized QueryPlan projection and scheduling context.

```rust
pub struct SearchRequest {
    pub search_request_id: String,
    pub query_plan_id: QueryPlanId,
    pub full_attempt_count: u8,
    pub retry_count: u8,
    pub search_round: u32,
    pub provider_id: ProviderId,
    pub query_text: String,
    pub semantic_description: String,
    pub max_results: u32,
    pub quality: QualityTier,
    pub quality_requirements: QualityRequirements,
    pub material_types: Vec<String>,
    pub visual_requirements: Vec<String>,
    pub negative_scope: Vec<String>,
    pub authorization_preference: AuthorizationPreference,
    pub provider_policy: QueryProviderPolicy,
    pub request_tags: BTreeMap<String, String>,
}
```

Rules:

- `query_text` comes from `NormalizedQueryPlan.query_texts`; if multiple query
  texts exist, the scheduler rotates or expands them across rounds.
- `max_results` is per provider invocation. It is derived from remaining
  candidate target, provider `max_results_per_request`, and a duplicate buffer.
- `full_attempt_count` and `retry_count` are read-only context from the
  orchestrator. `retry_count` must equal `full_attempt_count - 1`.
- No credential value may appear in this DTO.

```rust
    pub provider_raw_id: Option<String>,
    pub provider_result_url: Option<String>,

    pub search_query: String,
    pub search_round: u32,
    pub full_attempt_count: u8,
    pub retrieved_at: OffsetDateTime,

    pub license_evidence: LicenseEvidence,
    pub source_authority_hint: Option<String>,

```

Compatibility mapping:

- Current `CandidateRecord.id` becomes `candidate_id`.
- Current `source_url` becomes `image_url`.
- Current `page_url` becomes `source_page_url`.
- Current `dimensions` becomes `width` and `height`, or remains as an
  `ImageDimensions` subfield if development chooses that local style. Package
  serialization must expose width and height or an equivalent object.

Candidate construction invariants:

- `candidate_id` is deterministic within a run:
  `cand-{query_plan_id}-{stable_hash(provider_id, search_round, provider_raw_id or image_url or source_page_url)}`.
- `image_url` is required after normalization. Missing image URL is
  `CANDIDATE_IMAGE_URL_MISSING` and the record is not added as retrievable
  search candidate.
- `source_page_url` is optional but strongly preferred; missing value produces
  a warning for quality/retrieval.
- `license_hint` is evidence, not permission. Unknown license is not converted
  into "safe".
- `origin_candidate_ids` starts with the candidate's own ID and accumulates IDs
  of duplicates merged into this record.

### Dedupe Key

The scheduler computes a dedupe key with layered evidence:

1. Normalize direct `image_url`: lowercase host, strip common tracking query
   parameters, remove fragment, preserve meaningful path and filename.
2. Normalize `source_page_url` using the same URL normalization.
3. Include provider raw image identity when present.
4. Include dimensions only as weak evidence; dimensions alone are not a dedupe
   key.
5. Future artifact hash may strengthen dedupe after retrieval, but TASK-002
   does not require it.

```rust
pub struct CandidateDedupeEvidence {
    pub dedupe_key: String,
    pub image_url_key: Option<String>,
    pub source_page_key: Option<String>,
    pub provider_raw_key: Option<String>,
    pub weak_dimension_key: Option<String>,
    pub duplicate_of: Option<CandidateId>,
    pub merge_reason: DedupeMergeReason,
}
```

Merge rules:

- Exact normalized image URL match merges candidates.
- Exact provider raw ID match from the same provider merges candidates.
- Same source page plus same normalized image filename may merge.
- Same dimensions alone must not merge.
- If records merge, preserve the earliest/highest ranked candidate as the
  canonical record and append duplicate IDs to `origin_candidate_ids`.
- Provenance for merged records is retained in `CandidateProvenance` or an
  auxiliary `merged_provenance` list so package review can see every provider
  contribution.

## Control Flow

Search execution begins only after TASK-001 admission accepts the QueryPlan and
runtime config.

```text
NormalizedQueryPlan + RuntimeConfig.providers
  -> ProviderRegistry::from_config
  -> adapter lookup by provider_kind
  -> provider readiness reports
  -> effective weight table
  -> weighted scheduler loop
  -> provider SearchRequest
  -> provider SearchResponse
  -> candidate normalization
  -> dedupe and provenance merge
  -> SearchSessionOutcome
  -> TASK-003 candidate quality
```

Detailed sequence:

1. Orchestrator passes `NormalizedQueryPlan`, attempt counters, and
   `RuntimeConfig.providers` to the search module.
2. Registry constructs registrations and adapter map.
3. Registry evaluates readiness for all configured providers.
4. Registry builds `WeightedProviderEntry` rows from ready providers.
5. If the table is empty, return `SearchSessionOutcome` with
   `NO_AVAILABLE_SEARCH_PROVIDER`.
6. Scheduler initializes:
   - `candidate_target = NormalizedQueryPlan.candidate_target`
   - `search_round = 1`
   - `dedupe_index = empty`
   - `usage_events = empty`
   - `max_invocations = config limit or safe default`
7. While unique candidate count is below target:
   - select a provider by weighted random among non-exhausted providers
   - choose query text for the round
   - build `SearchRequest`
   - call `provider.search`
   - normalize response candidates
   - merge duplicates into `dedupe_index`
   - record `SearchUsageEvent`
   - mark provider exhausted if response is empty, next page absent, or adapter
     explicitly says exhausted
   - advance round or stop when target, exhaustion, or invocation cap occurs
8. Return `SearchSessionOutcome`.

The weighted scheduler must be deterministic in tests by accepting a
`RandomSource` implementation. Production may use an OS-backed or standard RNG,
but logs and package evidence must record selected provider and effective
weight for each invocation.

## Weighted Scheduling Algorithm

The effective weight table is:

```rust
pub struct WeightedProviderEntry {
    pub provider_id: ProviderId,
    pub display_name: String,
    pub effective_weight: u32,
    pub capabilities: ProviderConstraintSupport,
}
```

Algorithm:

```text
total_weight = sum(active_entries.effective_weight)
draw = rng.gen_range(total_weight)
for entry in active_entries:
  if draw < entry.effective_weight:
    return entry
  draw -= entry.effective_weight
```

Scheduling rules:

- Providers with status other than `Ready` are excluded.
- Providers with `effective_weight = 0` are excluded and diagnosed.
- Exhausted providers are temporarily removed from active entries for the
  current search session.
- Transient failures do not permanently remove a provider unless the adapter
  classifies the failure as terminal for the session.
- Each provider invocation requests enough results to cover the remaining target
  plus a duplicate buffer. A recommended formula is:
  `request_count = clamp(max(10, remaining_needed * 3), 1, provider_max_page_size)`.
- The scheduler stops when unique candidates reach `candidate_target`, all
  providers are exhausted, all ready providers are terminally failed, or the
  invocation cap is reached.

Shortage classification:

| Code | Meaning |
| --- | --- |
| `NO_AVAILABLE_SEARCH_PROVIDER` | No ready provider was in the effective table. |
| `ALL_PROVIDERS_EXHAUSTED` | Every ready provider signaled exhaustion before target. |
| `INSUFFICIENT_UNIQUE_CANDIDATES` | Raw recalls existed but dedupe left fewer than target. |
| `PROVIDER_PARTIAL_FAILURE` | Some provider failures reduced recall coverage. |
| `SEARCH_INVOCATION_LIMIT_REACHED` | Safety cap stopped scheduling before target. |

```json
  "diagnostics": [],
  "shortage_reason": "INSUFFICIENT_UNIQUE_CANDIDATES"
}
```

Package-safe output excludes credentials, raw response bodies, cookies, and
provider secrets.

## State Persistence And Invariants

TASK-002 introduces no durable storage outside later package materialization.
In-memory search state must be serializable for package handoff.

State fields:

```rust
pub struct SearchSessionState {
    pub query_plan_id: QueryPlanId,
    pub full_attempt_count: u8,
    pub retry_count: u8,
    pub search_round: u32,
    pub candidate_target: u32,
    pub unique_candidate_count: u32,
    pub invocation_count: u32,
    pub exhausted_providers: BTreeSet<ProviderId>,
    pub terminal_failed_providers: BTreeSet<ProviderId>,
    pub dedupe_index: BTreeMap<String, CandidateId>,
}
```

Invariants:

- `candidate_target` is copied from TASK-001 and equals
  `required_image_count * 20`.
- `retry_count = full_attempt_count - 1`.
- `candidate_id` uniqueness is scoped to a run and query plan.
- No candidate without a primary `image_url` enters the output candidates list.
- Every candidate belongs to exactly one `query_plan_id`.
- Every candidate has `provider_id`, `search_round`, `provider_rank`, and
  `dedupe_key`.
- Every duplicate merge preserves all origin IDs and provider provenance.
- Search can be short of target without failing the process, but shortage must
  be explicit and later coverage must account for it.

## Errors Diagnostics And Failure Taxonomy

Provider failures and search diagnostics must be machine-readable.

```rust
pub struct SearchDiagnostic {
    pub code: SearchDiagnosticCode,
    pub severity: DiagnosticSeverity,
    pub provider_id: Option<ProviderId>,
    pub candidate_id: Option<CandidateId>,
    pub search_request_id: Option<String>,
    pub message: String,
    pub remediation: Option<String>,
    pub redacted: bool,
}
```

Required diagnostic codes:

| Code | Severity | Meaning |
| --- | --- | --- |
| `DEFAULT_PROVIDER_UNAVAILABLE` | blocker | The configured default provider, normally `serpapi_google_images`, is unavailable and no policy-allowed fallback provider is ready. |
| `PROVIDER_DISABLED` | info | Provider configured but disabled. |
| `PROVIDER_ADAPTER_MISSING` | error | Configured provider kind has no adapter. |
| `PROVIDER_CREDENTIAL_MISSING` | error/blocker | Required credential env var is missing. |
| `PROVIDER_WEIGHT_INVALID` | error | Enabled provider has invalid scheduling weight. |
| `PROVIDER_HEALTH_FAILED` | warning/error | Provider health check failed. |
| `PROVIDER_QUOTA_EXHAUSTED` | warning/error | Provider quota or rate limit is exhausted. |
| `PROVIDER_CONSTRAINT_UNSUPPORTED` | warning/error | Query constraints cannot be supported by provider. |
| `PROVIDER_FIXTURE_NOT_PRODUCTION` | blocker | Fixture provider attempted in production mode. |
| `SEARCH_PROVIDER_TIMEOUT` | warning | Provider invocation timed out. |
| `SEARCH_PROVIDER_RATE_LIMITED` | warning | Provider request was rate limited. |
| `SEARCH_RESPONSE_UNNORMALIZABLE` | error | Provider response could not map to candidates. |
| `CANDIDATE_IMAGE_URL_MISSING` | warning/error | Provider result lacks direct image URL. |
| `CANDIDATE_SOURCE_URL_MISSING` | warning | Provider result lacks source page URL. |
| `CANDIDATE_DIMENSIONS_MISSING` | info | Provider did not return dimensions. |
| `CANDIDATE_LICENSE_UNKNOWN` | info | Provider did not return license hint. |
| `CANDIDATE_DUPLICATE_MERGED` | info | Candidate merged into an existing dedupe record. |
| `SEARCH_TARGET_SHORTAGE` | warning | Unique candidate target was not reached. |
| `NO_AVAILABLE_SEARCH_PROVIDER` | blocker | No ready provider is schedulable. |

Severity selection:

- Missing provider credentials are a blocker when the provider is required by
  policy or no other ready provider exists.
- Missing dimensions/license/source page are reference evidence, not automatic
  search blockers. TASK-003 may later block based on quality policy.
- Response normalization errors are adapter failures, not candidate failures.

## Security Authorization And Redaction

Security rules:

- Provider credentials are referenced by environment variable name from config.
- Resolved credential values stay inside provider adapter private state.
- `SearchRequest`, `SearchResponse`, `CandidateRecord`,
  `ProviderReadinessReport`, usage events, diagnostics, and package files must
  not contain API keys, bearer tokens, cookies, authorization headers, raw
  secrets, or private key text.
- Provider evidence may include endpoint host, status class, quota category, and
  credential env var name. It must not include full authenticated URLs if they
  contain tokens.
- Adapter logs and diagnostics must pass through the shared redaction helper
  defined by TASK-001 policy design.
- `license_hint` and `authorization_preference` are evidence for later policy
  and quality decisions. They do not grant usage rights.
- Explicitly prohibited provider kinds or domains must be reported as
  readiness blockers when configured.

Fixture boundary:

- `src/search/fixture.rs` remains test-only or explicit fixture mode.
- Fixture candidates must carry provider kind `fixture`.
- Production self-check reports fixture provider availability as non-production
  evidence only.
- A production `run` cannot reach `passed` using fixture search evidence.

  `origin_candidate_ids`.
