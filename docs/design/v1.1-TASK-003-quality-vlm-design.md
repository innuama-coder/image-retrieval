This document defines the v1.1 detailed design for candidate mechanical
validation, retrieved image mechanical acceptance, VLM subjective
evaluation, decision merging, audit evidence, diagnostics, and the handoff from
quality into artifact-backed retrieval planning.

This is documentation-only detailed design. It does not implement source code,
tests, Cargo manifests, CLI behavior, retrieval channels, or delivery package
writers.

## Scope And Boundary

### In Scope

- Candidate mechanical validation over normalized search candidates.
- Retrieved image mechanical acceptance over artifact-backed retrieval results.
- Blocking metric and reference metric taxonomy for both phases.
- Evidence DTOs, metric facts, decision DTOs, failure codes, audit events, and
  observability counters.
- `VlmEvaluationPort` v1.1 trait shape, request DTOs, response DTOs, and
  readiness behavior.
- Decision merging rules requiring both mechanical pass and VLM approval.
- Fail-closed behavior when production VLM is unavailable.
- Fixture-only evaluation boundaries for unit and integration tests.
- Handoff contract from accepted candidates into TASK-004 retrieval planning.

### Forbidden Scope And Non-Goals

- Do not implement Rust code in this task.
- Do not change source files, tests, manifests, or other design task files.
- Do not design retrieval channel execution; TASK-004 owns retrieval artifacts.
- Do not design the orchestrator retry loop or package writer; TASK-005 owns
  those details.
- Do not hard-code the production VLM endpoint or API token. The default
  provider is Qwen 3.5; endpoint/base URL, model id, and credential env var are
  runtime configuration values, with `QWEN_API_TOKEN` as the default token env
  var name.
- Do not allow fixture or mock subjective evaluation to satisfy production
  candidate or image acceptance.

## Current Source Delta

The existing code has a useful quality-gate foundation:

- `src/quality/candidate/mechanical.rs` already separates candidate blocking
  reasons from reference signals.
- `src/quality/candidate/evaluation.rs` defines candidate evaluation requests,
  conclusions, normalization, and VLM unexecutable semantics.
- `src/quality/candidate/gate.rs` runs mechanical validation before VLM and
  emits a `RetrievableCandidateSequence`.
- `src/quality/image/mechanical.rs` checks file size, content type, and
  dimensions for local images.
- `src/quality/image/evaluation.rs` mirrors candidate subjective evaluation for
  retrieved images.
- `src/quality/image/gate.rs` separates qualified and rejected images.
- `src/ports/mod.rs` exposes an `VlmEvaluationPort` trait.

The v1.1 gap is that these shapes are still MVP-oriented:

- `CandidateRecord` lacks the full TASK-002 fields: query ownership, source
  page URL naming, MIME type, license hint, dedupe key, origin IDs, and
  provenance.
- `ImageRecord` is path-only and cannot represent TASK-004 artifact-backed
  retrieval evidence such as sidecar, summary, task report, visual description,
  checksum, media match, fetch trace, and retrieval job ownership.
- Mechanical evidence uses strings in several places instead of typed,
  package-safe metric facts with evidence IDs and severities.
- `VlmEvaluationPort` currently accepts raw candidate/image lists plus a
  description, but v1.1 needs structured requests containing reference metrics,
  policy evidence, trace IDs, and redacted package-safe context.
- The production/fixture boundary exists in comments and tests but is not
  strong enough for release acceptance.

## Shared Quality Evidence Model

Candidate and image quality use the same evidence vocabulary so package
validation and audit tooling can reason over both phases.

```rust
pub enum QualityPhase {
    Candidate,
    Image,
}

pub enum MetricClass {
    Blocking,
    Reference,
}

pub enum QualitySeverity {
    Info,
    Warning,
    Error,
    Blocker,
}

pub struct MetricFact {
    pub metric_id: String,
    pub phase: QualityPhase,
    pub class: MetricClass,
    pub code: QualityMetricCode,
    pub severity: QualitySeverity,
    pub subject_id: String,
    pub query_plan_id: QueryPlanId,
    pub value: Option<String>,
    pub threshold: Option<String>,
    pub evidence_refs: Vec<String>,
    pub message: String,
    pub redacted: bool,
}
```

Rules:

- A non-empty blocking metric list means the subject fails mechanical
  validation and must not be sent to VLM.
- Reference metrics never accept or reject by themselves. They are evidence for
  VLM and later review unless policy explicitly promotes a metric to
  blocking.
- Every metric fact must be package-safe after redaction.
- Every metric fact must carry `query_plan_id` and `subject_id`; candidate
  facts use `candidate_id`, image facts use `retrieval_job_id` or local image
  artifact ID.
- Evidence refs point to package-safe in-memory or future package evidence
  paths, never secret-bearing raw provider/channel responses.

## Candidate Mechanical Validation

Candidate quality consumes TASK-002 `SearchSessionOutcome`. It must not treat a
search hit as suitable for retrieval until mechanical and VLM gates pass.

```rust
pub struct CandidateMechanicalAssessment {
    pub candidate_id: CandidateId,
    pub query_plan_id: QueryPlanId,
    pub passed: bool,
    pub blocking_metrics: Vec<MetricFact>,
    pub reference_metrics: Vec<MetricFact>,
    pub evaluated_at: OffsetDateTime,
}
```

### Candidate Blocking Metrics

| Code | Blocking condition |
| --- | --- |
| `CANDIDATE_IMAGE_URL_MISSING` | No primary `image_url` is present after normalization. |
| `CANDIDATE_IMAGE_URL_INVALID` | URL is empty, malformed, unsupported, or non-HTTP(S). |
| `CANDIDATE_QUERY_OWNERSHIP_MISMATCH` | Candidate `query_plan_id` differs from the active QueryPlan. |
| `CANDIDATE_DUPLICATE_BLOCKED` | Candidate is an exact duplicate already represented by a higher-priority candidate. |
| `CANDIDATE_PROHIBITED_SOURCE` | Provider/source domain is prohibited by policy. |
| `CANDIDATE_ACCESS_RESTRICTED` | Candidate is known login-required/paywalled and policy disallows it. |
| `CANDIDATE_NEGATIVE_SCOPE_CONTRADICTION` | Title/snippet/source evidence clearly contradicts `negative_scope` or `must_avoid`. |
| `CANDIDATE_BELOW_ABSOLUTE_DIMENSIONS` | Provider dimensions are present and below absolute image minimums. |
| `CANDIDATE_FIXTURE_NOT_PRODUCTION` | Fixture candidate evidence is used in production mode. |

### Candidate Reference Metrics

| Code | Reference evidence |
| --- | --- |
| `CANDIDATE_PROVIDER_RANK` | Provider rank and global rank hint. |
| `CANDIDATE_DIMENSIONS_REPORTED` | Width/height or missing dimension evidence. |
| `CANDIDATE_SOURCE_PAGE_PRESENT` | Whether source page URL exists and is usable by retrieval. |
| `CANDIDATE_LICENSE_HINT` | License or authorization hint, including unknown. |
| `CANDIDATE_SOURCE_AUTHORITY_HINT` | Source authority, domain, or provider evidence. |
| `CANDIDATE_TEXT_CONTEXT_MATCH` | Title/snippet match and missing-context signals. |
| `CANDIDATE_DIVERSITY_SIGNAL` | Source/provider diversity and duplicate similarity. |
| `CANDIDATE_SEARCH_SHORTAGE_CONTEXT` | Search target shortage evidence from TASK-002. |

Implementation note for development: current `CandidateBlockingReason` and
`CandidateReferenceSignal` can be migrated into these metric facts rather than
deleted immediately. Public v1.1 package output must use the metric-fact shape.

## Candidate VLM Evaluation

Production candidate subjective evaluation must call the configured VLM
provider. The v1.1 default provider is Qwen 3.5. Mechanically blocked
candidates do not reach VLM. Mechanically passed candidates enter VLM as a
batch.

```rust
pub struct VlmCandidateEvaluationRequest {
    pub request_id: String,
    pub query_plan_id: QueryPlanId,
    pub full_attempt_count: u8,
    pub retry_count: u8,
    pub semantic_description: String,
    pub quality: QualityTier,
    pub quality_requirements: QualityRequirements,
    pub visual_requirements: Vec<String>,
    pub negative_scope: Vec<String>,
    pub candidates: Vec<CandidateEvaluationSubject>,
    pub policy_context: QualityPolicyContext,
    pub model: String,
    pub evaluator_provider_id: String,
    pub fixture_mode: bool,
}

pub struct CandidateEvaluationSubject {
    pub candidate: CandidateRecord,
    pub mechanical_assessment: CandidateMechanicalAssessment,
    pub reference_metrics: Vec<MetricFact>,
}
```

Security rules:

- Request DTOs must not contain provider credentials, cookies, authorization
  headers, raw VLM credentials, or full authenticated URLs.
- URLs are allowed only after redaction strips secret query parameters.
- `fixture_mode = true` is allowed only under test or explicit fixture runs and
  cannot satisfy production delivery evidence.

VLM response:

```rust
pub struct VlmEvaluationResponse {
    pub request_id: String,
    pub evaluator_id: String,
    pub evaluator_kind: VlmEvaluatorKind,
    pub status: VlmResponseStatus,
    pub decisions: Vec<VlmSubjectDecision>,
    pub diagnostics: Vec<QualityDiagnostic>,
    pub audit_ref: Option<String>,
    pub redaction_applied: bool,
}

pub enum VlmSubjectDecisionKind {
    Approve,
    Reject,
    Uncertain,
    Unexecutable,
}

pub struct VlmSubjectDecision {
    pub subject_id: String,
    pub decision: VlmSubjectDecisionKind,
    pub confidence: Option<f32>,
    pub reason_codes: Vec<String>,
    pub rationale_summary: String,
    pub evidence_refs: Vec<String>,
}
```

Response invariants:

- There must be exactly one decision for each submitted subject.
- Unknown subject IDs are diagnostic code `VLM_EVALUATION_RESPONSE_UNKNOWN_SUBJECT`.
- Missing subject decisions are diagnostic code
  `VLM_EVALUATION_RESPONSE_MISSING_DECISION`.
- `Uncertain` is not accepted.
- `Unexecutable` is an execution block, not a rejection.

## Candidate Decision Merging And Handoff

Candidate acceptance requires both:

1. `CandidateMechanicalAssessment.passed = true`
2. `VlmSubjectDecision.decision = Approve`

```rust
pub struct CandidateQualityDecision {
    pub candidate_id: CandidateId,
    pub query_plan_id: QueryPlanId,
    pub mechanical_passed: bool,
    pub vlm_passed: bool,
    pub final_status: CandidateQualityStatus,
    pub priority: u32,
    pub blocking_metrics: Vec<MetricFact>,
    pub reference_metrics: Vec<MetricFact>,
    pub vlm_decision: Option<VlmSubjectDecision>,
    pub diagnostics: Vec<QualityDiagnostic>,
}

pub enum CandidateQualityStatus {
    Retrievable,
    MechanicallyRejected,
    SubjectivelyRejected,
    SubjectivelyUncertain,
    ExecutionBlocked,
}
```

Decision mapping:

| Mechanical | VLM | Final status | TASK-004 eligible |
| --- | --- | --- | --- |
| fail | not called | `MechanicallyRejected` | no |
| pass | approve | `Retrievable` | yes |
| pass | reject | `SubjectivelyRejected` | no |
| pass | uncertain | `SubjectivelyUncertain` | no |
| pass | unavailable/unexecutable | `ExecutionBlocked` | no |

The TASK-004 handoff is:

```rust
pub struct RetrievableCandidateBatch {
    pub query_plan_id: QueryPlanId,
    pub full_attempt_count: u8,
    pub retry_count: u8,
    pub retrieval_batch_target: u32,
    pub candidates: Vec<RetrievableCandidate>,
    pub rejected_decisions: Vec<CandidateQualityDecision>,
    pub execution_blocking_facts: Vec<QualityExecutionBlock>,
}

pub struct RetrievableCandidate {
    pub candidate: CandidateRecord,
    pub candidate_quality_decision: CandidateQualityDecision,
    pub retrieval_priority: u32,
    pub primary_image_url: String,
    pub source_page_url: Option<String>,
    pub thumbnail_url: Option<String>,
    pub expected_mime_type: Option<String>,
    pub license_hint: Option<String>,
    pub provenance_refs: Vec<String>,
}
```

Rules:

- `RetrievableCandidateBatch.candidates.len()` may exceed or fall below
  `retrieval_batch_target`; TASK-004 selects at most the configured batch target
  for a retrieval attempt.
- Thumbnail URL must not become the primary retrieval target for delivery unless
  a later explicit non-delivery evidence role is introduced.
- The batch must preserve rejected decisions so TASK-005 can produce manifest
  and coverage gap explanations.

Delivery qualification rules:

- `delivered_qualified = true` only for `DeliveredQualified`.
- Metadata-only, thumbnail-only, source-page-only, sidecar-only, summary-only,
  or task-report-only results are never delivered images.
- VLM approval cannot make a mechanically rejected image delivered.
- Mechanical pass cannot make a VLM rejected/uncertain image delivered.
- Execution-blocked images are counted as gaps, not accepted images.

## VLM Port Contract

The v1.1 port should be structured and readiness-aware:

```rust
pub trait VlmEvaluationPort: Send + Sync {
    fn readiness(&self, config: &VlmEvaluationConfig) -> VlmEvaluationReadinessReport;

    fn evaluate_candidates(
        &self,
        request: &VlmCandidateEvaluationRequest,
    ) -> Result<VlmEvaluationResponse, VlmEvaluationError>;

    fn evaluate_images(
        &self,
        request: &VlmImageEvaluationRequest,
    ) -> Result<VlmEvaluationResponse, VlmEvaluationError>;
}
```

Default Qwen 3.5 configuration:

```toml
[vlm_evaluation]
provider_id = "qwen_3_5_vlm"
provider_kind = "qwen_3_5_vlm"
model = "qwen-3.5"
credential_env = "QWEN_API_TOKEN"
endpoint = "<configured externally>"
enabled = true
```

Rules:

- The resolved `QWEN_API_TOKEN` value is private to the adapter and must never
  appear in requests saved to package evidence, diagnostics, logs, or reports.
- Runtime config may override the credential env var name, endpoint/base URL,
  timeout, and model id.
- The adapter must submit candidate reference metrics as text context and
  retrieved image local artifacts as image inputs where the provider API
  supports it.

Readiness:

```rust
pub struct VlmEvaluationReadinessReport {
    pub enabled: bool,
    pub available: bool,
    pub fixture_mode: bool,
    pub endpoint_configured: bool,
    pub candidate_prompt_configured: bool,
    pub image_prompt_configured: bool,
    pub credential_status: CredentialStatus,
    pub failure_code: Option<VlmEvaluationFailureCode>,
    pub checked_at: OffsetDateTime,
    pub evidence: Vec<String>,
    pub redaction_applied: bool,
}
```

Required failure codes:

| Code | Meaning |
| --- | --- |
| `VLM_EVALUATION_DISABLED` | VLM is disabled in config. |
| `VLM_EVALUATION_ENDPOINT_MISSING` | Production endpoint or base URL is absent. |
| `VLM_EVALUATION_CREDENTIAL_MISSING` | Required credential env var is absent. |
| `VLM_EVALUATION_CANDIDATE_PROMPT_MISSING` | Candidate evaluation prompt/template is absent. |
| `VLM_EVALUATION_IMAGE_PROMPT_MISSING` | Image evaluation prompt/template is absent. |
| `VLM_EVALUATION_HEALTH_FAILED` | Health/readiness probe failed. |
| `VLM_EVALUATION_TIMEOUT` | Evaluation request timed out. |
| `VLM_EVALUATION_RESPONSE_INVALID` | Response could not be validated. |
| `VLM_EVALUATION_FIXTURE_NOT_PRODUCTION` | Fixture evaluator attempted in production mode. |
| `VLM_EVALUATION_UNAVAILABLE` | General production unavailability. |

Production behavior:

- If readiness is not available and any mechanically passed subject requires
  subjective evaluation, the quality phase returns `ExecutionBlocked`.
- The orchestrator may retry the full workflow only under TASK-005 retry rules,
  but the current attempt cannot treat missing VLM as subjective reject or
  success.
- Fixture subjective evaluation is allowed only under test/fixture execution and
  must be marked `fixture_mode = true` in diagnostics and audit events.

## Control Flow

Candidate phase:

```text
SearchSessionOutcome
  -> validate candidate ownership and policy context
  -> candidate mechanical assessment for each candidate
  -> split mechanically blocked vs mechanically passed
  -> if mechanically passed is empty: return no retrievable candidates
  -> VLM readiness check
  -> if production VLM unavailable: execution_blocked
  -> build VlmCandidateEvaluationRequest
  -> call VlmEvaluationPort::evaluate_candidates
  -> validate response cardinality and subject ids
  -> merge mechanical + VLM decisions
  -> produce RetrievableCandidateBatch for TASK-004
```

Image phase:

```text
RetrievalBatchResult
  -> validate retrieval job ownership and artifact completeness
  -> image mechanical assessment for each retrieval result
  -> split mechanically blocked vs mechanically passed
  -> if mechanically passed is empty: return no accepted images
  -> VLM readiness check
  -> if production VLM unavailable: execution_blocked
  -> build VlmImageEvaluationRequest
  -> call VlmEvaluationPort::evaluate_images
  -> validate response cardinality and subject ids
  -> merge mechanical + VLM decisions
  -> produce ImageAcceptanceOutcome for TASK-005
```

## State Persistence And Invariants

Quality state is in-memory until TASK-005 package materialization. It must be
serializable and package-safe.

Invariants:

- `retry_count = full_attempt_count - 1`.
- Candidate quality decisions retain `query_plan_id` and `candidate_id`.
- Image acceptance decisions retain `query_plan_id`, `candidate_id`, and
  `retrieval_job_id`.
- Mechanically blocked subjects do not reach VLM.
- Every VLM request has a response decision for every submitted subject or
  the whole response is invalid.
- Production VLM unavailable produces `ExecutionBlocked`.
- `delivered_qualified = true` implies mechanical pass, VLM pass, artifact
  refs present, ownership match, and no negative-scope blocker.

## Errors Diagnostics And Failure Taxonomy

```rust
pub struct QualityDiagnostic {
    pub code: QualityDiagnosticCode,
    pub severity: QualitySeverity,
    pub phase: QualityPhase,
    pub subject_id: Option<String>,
    pub query_plan_id: QueryPlanId,
    pub message: String,
    pub remediation: Option<String>,
    pub evidence_refs: Vec<String>,
    pub redacted: bool,
}
```

Required diagnostic codes:

| Code | Severity | Meaning |
| --- | --- | --- |
| `QUALITY_CANDIDATE_MECHANICAL_BLOCK` | info/warning | Candidate failed a deterministic blocking metric. |
| `QUALITY_IMAGE_MECHANICAL_BLOCK` | warning/error | Retrieved image failed deterministic artifact or image checks. |
| `QUALITY_VLM_EVALUATION_UNAVAILABLE` | blocker | Production VLM cannot run. |
| `QUALITY_VLM_EVALUATION_FIXTURE_NOT_PRODUCTION` | blocker | Fixture evaluator attempted in production. |
| `QUALITY_VLM_EVALUATION_TIMEOUT` | blocker/retryable | VLM request timed out. |
| `QUALITY_VLM_EVALUATION_RESPONSE_INVALID` | blocker | VLM response has invalid schema/cardinality/subject IDs. |
| `QUALITY_SUBJECTIVE_REJECTED` | info | VLM rejected subject. |
| `QUALITY_SUBJECTIVE_UNCERTAIN` | info | VLM could not approve subject. |
| `QUALITY_ARTIFACT_EVIDENCE_MISSING` | error | Image phase required artifact evidence is missing. |
| `QUALITY_OWNERSHIP_MISMATCH` | error | Candidate/query/retrieval ownership mismatch. |
| `QUALITY_NEGATIVE_SCOPE_BLOCK` | error | Negative scope contradiction blocks candidate or image. |
| `QUALITY_SENSITIVE_DATA_REDACTED` | warning | Redaction changed request/response/diagnostic text. |

Severity rules:

- VLM unavailable in production is a blocker.
- Candidate mechanical block is usually an item-level rejection, not a run
  blocker, unless all candidates are blocked and no retry can produce more.
- Image artifact evidence missing is item-level rejection and package validation
  failure if the item is later marked delivered.
- Invalid VLM response blocks the subjective evaluation phase because the
  system cannot safely infer decisions.

## Security Redaction And Authorization

Security rules:

- No VLM request, response, metric fact, diagnostic, audit event, or
  package-ready quality output may contain resolved provider/channel/VLM
  credentials.
- Sensitive query parameters in URLs must be stripped or redacted before
  VLM submission unless VLM requires the URL for public fetch context;
  local artifact paths are preferred for image evaluation.
- Cookies, authorization headers, bearer tokens, API keys, private keys, and
  raw authenticated response bodies are forbidden in quality DTOs.
- VLM rationale is stored as a summary, not full unrestricted transcript,
  unless a future policy explicitly allows richer audit capture.
- Explicit prohibited source evidence is blocking. Unknown authorization remains
  reference evidence unless policy makes it blocking.
- Fixture flags must be carried through diagnostics and cannot be hidden by
  redaction.

## Fixture Only Vs Production Boundary

Fixture behavior is required for deterministic tests but cannot satisfy
production product goals.

Rules:

- Fixture candidate/image evaluators must live behind explicit fixture mode.
- Fixture mode outputs must include `fixture_mode = true`,
  `evaluator_kind = Fixture`, and diagnostic
  `QUALITY_VLM_EVALUATION_FIXTURE_NOT_PRODUCTION` when used in production context.
- Production self-check must report fixture VLM as not production-ready.
- Production `run` must not mark a candidate as `Retrievable` or an image as
  `DeliveredQualified` using fixture subjective evaluation.
- Test helpers such as current `evaluate_with_conclusions` and
  `evaluate_images_with_conclusions` remain valid for unit tests only.

## Verification Plan

Unit tests:

- Candidate with missing or invalid `image_url` is mechanically blocked.
- Candidate with query ownership mismatch is mechanically blocked.
- Candidate with prohibited source is mechanically blocked.
- Candidate negative-scope contradiction is mechanically blocked.
- Missing dimensions/license/source page produce reference metrics, not
  automatic blocking.
- Mechanically blocked candidates are not submitted to VLM.
- VLM approve maps to `Retrievable`.
- VLM reject maps to `SubjectivelyRejected`.
- VLM uncertain maps to `SubjectivelyUncertain`.
- VLM unexecutable maps to `ExecutionBlocked`.
- Candidate response cardinality mismatch produces
  `QUALITY_VLM_EVALUATION_RESPONSE_INVALID`.
- Fixture evaluator in production produces
  `QUALITY_VLM_EVALUATION_FIXTURE_NOT_PRODUCTION`.
- Image missing local artifact, source artifact, sidecar, summary, task report,
  visual description, checksum, or media match is mechanically blocked.
- Metadata-only retrieval result is mechanically blocked.
- VLM image approve plus mechanical pass maps to
  `DeliveredQualified`.
- VLM image reject/uncertain does not count as delivered.
- Redaction removes credential-like values from metrics, diagnostics, and
  VLM request summaries.

Integration tests:

- `SearchSessionOutcome` from TASK-002 flows into `CandidateQualityOutcome`
  with retained candidate provenance.
- `RetrievableCandidateBatch` includes only candidates passing both candidate
  mechanical and VLM gates.
- VLM unavailable in production blocks candidate quality when there are
  mechanically passed candidates.
- `RetrievalBatchResult` from TASK-004 flows into `ImageAcceptanceOutcome` with
  retained retrieval job and artifact refs.
- Artifact-backed image acceptance rejects missing sidecar/summary/report before
  VLM submission.
- VLM unavailable in production blocks image acceptance when there are
  mechanically passed images.
- Fixture mode drives deterministic tests but cannot produce production
  package-ready decisions.

E2E and package validation tests:

- A fixture E2E run can exercise all quality states under explicit fixture mode.
- A production run with VLM unavailable cannot reach `passed`.
- A delivered image in the package has candidate mechanical evidence, candidate
  VLM decision, image mechanical evidence, image VLM decision, and all
  artifact refs.
- Package validation fails if any delivered image lacks VLM candidate or
  image decision evidence.
