This document defines the v1.1 detailed design for retrieval jobs, retrieval
batches, artifact-backed channel results, fallback execution, sidecars, content
summaries, task reports, visual descriptions, checksums, content-type sniffing,
job ownership, policy blockers, diagnostics, observability, and handoff into
TASK-003 image acceptance and TASK-005 package validation.

This is documentation-only detailed design. It does not implement code, tests,
Cargo manifest changes, or delivery package writers.

## Scope And Boundary

### In Scope

- `RetrievalChannel` / `BaseRetrievalChannel` v1.1 trait migration.
- `RetrievalJob`, `RetrievalBatch`, `RetrievalArtifactResult`,
  `RetrievalBatchResult`, channel readiness, and attempt trace DTOs.
- Batch target consumption from TASK-001:
  `NormalizedQueryPlan.retrieval_batch_target = required_image_count * 2`.
- Planning retrieval jobs from TASK-003 `RetrievableCandidateBatch`.
- Normal web fetch tier with direct image fetch and source-page resolve
  attempts.
- Self-hosted service channel boundary for open-source/self-hosted retrieval.
- Paid online service channel boundary, disabled by default and policy gated.
- Artifact paths: local image artifact, source artifact, source sidecar,
  content summary, task report, visual description, diagnostics, and trace.
- Checksum, file size, content-type sniffing, media-type match, dimensions,
  metadata-only rejection, and partial/fallback diagnostics.
- Robots/site-rule, login, paywall, authorization, and prohibited-source
  blockers.
- Observability, metrics, audit events, redaction, and package-safe handoff.
- Verification matrix for unit, integration, fixture, package, and blocker
  evidence.

### Forbidden Scope And Non-Goals

- Do not implement code in this task.
- Do not modify source, tests, Cargo manifests, fixtures, or other design
  documents.
- Do not mark images as delivered. Retrieval produces artifact evidence only.
- Do not bypass robots, authorization, login, paywall, or access restrictions
  by escalating to higher tiers.
- Do not silently enable paid channels.
- Do not treat metadata-only, thumbnail-only, source-page-only, summary-only,
  sidecar-only, or task-report-only results as retrieved images.
- Do not use fixture retrieval artifacts as production delivery evidence.
- Do not decide the final robots/site-rule product posture; this design models
  warning/blocker behavior through policy.

## Current Source Delta

The existing retrieval source is a useful MVP foundation:

- `src/domain/retrieval.rs` defines `RetrievalChannelTier`,
  `RetrievalBatch`, `RetrievalResult`, `RetrievalSuccess`,
  `RetrievalFailure`, `FallbackEligibilityFact`, `ChannelAttemptResult`, and
  `RetrievalOutcome`.
- `src/retrieval/batch_planner.rs` already derives
  `batch_target = required_count * 2`, selects candidates by priority, and
  records short-batch evidence.
- `src/retrieval/mod.rs` executes a single batch through ordered channels,
  records fallback facts, blocks access-restricted fallback, and detects paid
  unconfirmed readiness.
- `src/retrieval/channels/web_fetch.rs` implements a minimal HTTP GET channel
  that writes local bytes and checks server `Content-Type`.
- `src/retrieval/channels/fixture.rs` provides deterministic channel behavior
  for tests.
- `src/ports/mod.rs` exposes `BaseRetrievalChannel` as a trait boundary.

The v1.1 gap is that the current model is path-oriented and batch-oriented, not
artifact-complete:

- `RetrievalSuccess` has only `candidate_id`, `local_path`, `channel_tier`,
  `content_type`, and `file_size_bytes`.
- It lacks `retrieval_job_id`, `query_plan_id`, `channel_id`, source artifact,
  sidecar, content summary, task report, visual description, checksum,
  dimensions, media-type sniff evidence, attempt trace, policy decisions,
  redaction status, and package evidence references.
- `BaseRetrievalChannel::retrieve_batch` returns `Vec<RetrievalResult>` rather
  than a structured `RetrievalBatchResult`.
- Direct web fetch success does not yet create the same evidence set expected
  from content-retrieval style workflows.
- Source-page resolve is not a separate attempt type.
- Self-hosted and paid channel boundaries exist only as tiers/fixtures, not as
  readiness-aware adapters.

## Interfaces Types And DTO Contracts

### Channel Identity And Tiers

The serialized tier names should migrate from MVP names to v1.1 canonical
names while preserving input aliases where useful:

```rust
pub enum RetrievalChannelTier {
    #[serde(rename = "normal_web_fetch", alias = "web_fetch")]
    NormalWebFetch,
    #[serde(rename = "self_hosted_service", alias = "self_hosted")]
    SelfHostedService,
    #[serde(rename = "paid_online_service", alias = "paid")]
    PaidOnlineService,
}
```

Tier order is fixed:

1. `normal_web_fetch`
2. `self_hosted_service`
3. `paid_online_service`

Within `normal_web_fetch`, attempts are ordered:

1. `direct_image_fetch`
2. `source_page_resolve`

These are attempt modes, not separate paid or higher tiers. They must be
recorded separately because direct image fetch and source-page resolve have
different failure modes and audit evidence.

### Canonical Retrieval Channel Trait

`BaseRetrievalChannel` remains the canonical port name in Rust. The product
language may also call it `RetrievalChannel`. New implementations should use
the v1.1 contract:

```rust
pub trait BaseRetrievalChannel: Send + Sync {
    fn channel_id(&self) -> RetrievalChannelId;
    fn display_name(&self) -> &str;
    fn tier(&self) -> RetrievalChannelTier;
    fn capabilities(&self) -> RetrievalChannelCapabilities;
    fn readiness(&self, config: &RetrievalChannelConfig) -> RetrievalChannelReadinessReport;
    fn retrieve_batch(
        &self,
        batch: &RetrievalBatch,
    ) -> Result<RetrievalBatchResult, RetrievalError>;
}
```

Migration rule:

- Existing `BaseRetrievalChannel::readiness() -> Result<()>` and
  `retrieve_batch() -> Vec<RetrievalResult>` can remain behind compatibility
  wrappers for old tests.
- The public v1.1 path used by orchestrator, self-check, quality, delivery,
  and validation must use `RetrievalBatchResult`.
- Fixture channels implement the new trait only under explicit fixture/test
  config and must return fixture diagnostics when used in production mode.

### Channel Capabilities

```rust
pub struct RetrievalChannelCapabilities {
    pub supports_direct_image_fetch: bool,
    pub supports_source_page_resolve: bool,
    pub supports_sidecar: bool,
    pub supports_content_summary: bool,
    pub supports_task_report: bool,
    pub supports_visual_description: bool,
    pub supports_checksum: bool,
    pub supports_content_type_sniffing: bool,
    pub supports_dimension_probe: bool,
    pub supports_batching: bool,
    pub max_batch_size: Option<u32>,
    pub fixture_only: bool,
}
```

Capability rules:

- A channel may be ready for a fallback role only if it can produce the full
  required evidence set or the wrapper can synthesize missing evidence locally.
- `normal_web_fetch` may synthesize sidecar, content summary, task report, and
  visual description from the fetched artifact and HTTP metadata.
- If a self-hosted or paid service returns only a URL or metadata, the adapter
  must map it to `RetrievalStatus::MetadataOnlyRejected`, not success.
- Fixture capabilities do not satisfy production readiness.

### Channel Readiness Report

```rust
pub struct RetrievalChannelReadinessReport {
    pub channel_id: RetrievalChannelId,
    pub display_name: String,
    pub tier: RetrievalChannelTier,
    pub enabled: bool,
    pub available: bool,
    pub included_in_fallback_order: bool,
    pub credential_status: CredentialStatus,
    pub dependency_status: DependencyStatus,
    pub policy_status: RetrievalPolicyStatus,
    pub failure_code: Option<RetrievalFailureCode>,
    pub checked_at: OffsetDateTime,
    pub evidence: Vec<RetrievalEvidenceFact>,
    pub redaction_applied: bool,
}
```

Required readiness failure codes:

| Code | Meaning |
| --- | --- |
| `RETRIEVAL_CHANNEL_DISABLED` | Config disables the channel. |
| `RETRIEVAL_CHANNEL_ADAPTER_MISSING` | Config names a kind without an adapter. |
| `RETRIEVAL_CHANNEL_DEPENDENCY_MISSING` | Local binary, library, writable directory, or service endpoint is missing. |
| `RETRIEVAL_CHANNEL_MISCONFIGURED` | Endpoint, output path, timeout, or max batch config is invalid. |
| `RETRIEVAL_CHANNEL_CREDENTIAL_MISSING` | Required credential env var is absent. |
| `RETRIEVAL_PAID_UNCONFIRMED` | Paid channel lacks explicit runtime and query approval. |
| `RETRIEVAL_ROBOTS_POLICY_UNDECIDED` | Robots/site-rule behavior is not closed under current policy. |
| `RETRIEVAL_FIXTURE_NOT_PRODUCTION` | Fixture channel was configured for a production run. |
| `RETRIEVAL_UNAVAILABLE` | Unknown or transient channel unavailability. |

Readiness is package-safe. It may expose env var names, endpoint host, channel
kind, and dependency categories. It must not expose resolved tokens, cookies,
authorization headers, private keys, or authenticated URLs.

### Retrieval Target

TASK-004 consumes TASK-003 `RetrievableCandidate`. A retrieval target preserves
search provenance and candidate quality evidence, but it does not re-run
candidate quality.

```rust
pub struct RetrievalTarget {
    pub target_type: RetrievalTargetType,
    pub primary_image_url: String,
    pub alternate_source_page_url: Option<String>,
    pub thumbnail_url: Option<String>,
    pub expected_mime_type: Option<String>,
    pub license_hint: Option<String>,
    pub provider_id: ProviderId,
    pub candidate_provenance_refs: Vec<String>,
}

pub enum RetrievalTargetType {
    Image,
}
```

Rules:

- `primary_image_url` comes from TASK-003 `RetrievableCandidate.primary_image_url`.
- `alternate_source_page_url` comes from the candidate source page URL and is
  used only for `source_page_resolve` fallback.
- `thumbnail_url` is reference evidence only. It must not become the primary
  delivery image target unless a future non-delivery role is explicitly added.
- `expected_mime_type` is evidence, not proof. Actual content must be sniffed.

### Retrieval Job

Each selected retrievable candidate becomes one `RetrievalJob` for the current
full attempt:

```rust
pub struct RetrievalJob {
    pub retrieval_job_id: RetrievalJobId,
    pub query_plan_id: QueryPlanId,
    pub candidate_id: CandidateId,
    pub full_attempt_count: u8,
    pub retry_count: u8,
    pub retrieval_priority: u32,
    pub target: RetrievalTarget,
    pub candidate_quality_decision_ref: String,
    pub requested_outputs: Vec<RequestedRetrievalOutput>,
    pub policy_context: RetrievalPolicyContext,
}

pub enum RequestedRetrievalOutput {
    LocalArtifact,
    SourceArtifact,
    SourceSidecar,
    ContentSummary,
    TaskReport,
    VisualDescription,
    Checksum,
    FetchTrace,
}
```

Job invariants:

- `retrieval_job_id` is unique within a run and stable enough for package
  links, for example `ret-{query_plan_id}-{full_attempt_count}-{candidate_id}`.
- `query_plan_id` and `candidate_id` must match the input
  `RetrievableCandidate`.
- `retry_count = full_attempt_count - 1`.
- `requested_outputs` must include every enum value listed above for production
  retrieval.
- Jobs that cannot satisfy policy before network execution become
  `RetrievalStatus::PolicyBlocked`.

### Retrieval Batch

`RetrievalBatch` should migrate from IDs plus URL map to structured jobs:

```rust
pub struct RetrievalBatch {
    pub retrieval_batch_id: String,
    pub query_plan_id: QueryPlanId,
    pub full_attempt_count: u8,
    pub retry_count: u8,
    pub target_size: u32,
    pub actual_size: u32,
    pub is_short_batch: bool,
    pub jobs: Vec<RetrievalJob>,
    pub shortage: Option<RetrievalBatchShortage>,
}
```

Batch planning rules:

- `target_size = NormalizedQueryPlan.retrieval_batch_target`.
- By default, `target_size = required_image_count * 2`.
- Take at most `target_size` candidates from TASK-003
  `RetrievableCandidateBatch.candidates`, sorted by `retrieval_priority`.
- If fewer candidates exist, create a short batch and include shortage evidence.
- Rejected, uncertain, mechanically blocked, or Qwen 3.5 VLM execution-blocked
  candidates do not become jobs.
- Already accepted images from previous full attempts must be excluded by
  TASK-005 before TASK-004 batch planning is invoked.
- Previously failed candidates may be excluded or retried only when TASK-005
  supplies policy that a different fallback path or changed policy is allowed.

- `INSUFFICIENT_RETRIEVABLE_CANDIDATES`

## Artifact Result Contract

### Retrieval Status

```rust
pub enum RetrievalStatus {
    Complete,
    Partial,
    Failed,
    PolicyBlocked,
    AccessRestricted,
    PaidUnconfirmed,
    MetadataOnlyRejected,
    FixtureRejectedForProduction,
}
```

Status rules:

- `Complete` means all required artifact fields exist and integrity evidence
  was produced. It does not mean the image is delivered.
- `Partial` means at least one artifact exists but required evidence is
  incomplete. Partial results proceed to TASK-003 image mechanical acceptance
  only as rejected evidence, not as delivered candidates.
- `Failed` means no usable artifact was produced.
- `PolicyBlocked`, `AccessRestricted`, and `PaidUnconfirmed` are explicit
  blocker statuses with diagnostic evidence.
- `MetadataOnlyRejected` is mandatory when a channel returns only a URL,
  source page, HTML, metadata, summary, or remote reference.
- `FixtureRejectedForProduction` is mandatory when fixture evidence appears in
  production mode.


Required completeness for `RetrievalStatus::Complete`:

- `local_artifact_path` present and file exists.
- `source_artifact_path` present and file exists.
- `source_sidecar_path` present and sidecar schema valid.
- `content_summary_path` present and summary quality gate passed.
- `task_report_path` present and task report valid.
- `visual_description_path` present and visual description valid.
- `checksum_sha256` present.
- `content_type` present after reported/sniffed reconciliation.
- `file_size_bytes > 0`.
- `image_dimensions` present for raster formats where probe is supported; for
  SVG or unsupported formats a diagnostic explains dimension-probe status.
- `media_type_match = true`.
- `job_ownership_valid = true`.
- `metadata_only = false`.

If any completeness field is missing, status is `Partial` or a more specific
failure status, never `Complete`.

### Retrieval Batch Result

```rust
pub struct RetrievalBatchResult {
    pub retrieval_batch_id: String,
    pub query_plan_id: QueryPlanId,
    pub full_attempt_count: u8,
    pub retry_count: u8,
    pub target_size: u32,
    pub actual_size: u32,
    pub channel_readiness: Vec<RetrievalChannelReadinessReport>,
    pub results: Vec<RetrievalArtifactResult>,
    pub attempt_trace: Vec<RetrievalAttemptTrace>,
    pub fallback_decisions: Vec<RetrievalFallbackDecision>,
    pub shortage: Option<RetrievalBatchShortage>,
    pub execution_blocking_facts: Vec<RetrievalExecutionBlock>,
    pub diagnostics: Vec<RetrievalDiagnostic>,
    pub summary: RetrievalBatchSummary,
}
```

Result invariants:

- There is one terminal `RetrievalArtifactResult` for every job in
  `RetrievalBatch.jobs`.
- A job may have multiple attempts in `fetch_trace`, but exactly one terminal
  artifact result.
- Batch-level `attempt_trace` is the union of per-result traces plus abandoned
  channel attempts.
- `retry_count = full_attempt_count - 1`.
- `target_size` equals the batch target from TASK-001.
- `actual_size = jobs.len()`.
- Results are artifact evidence only. TASK-003 image acceptance and TASK-005
  package validation decide delivered status.

## Required Artifact Files

Retrieval writes into a run-owned staging area later copied or linked into the
canonical package by TASK-005:

```text
staging/retrieval/
  attempts/{full_attempt_count}/
    {retrieval_job_id}/
      artifact.{ext}
      source-artifact.{ext}
      source-sidecar.json
      content-summary.json
      task-report.json
      visual-description.json
      diagnostics.json
      fetch-trace.json
```

`local_artifact_path` and `source_artifact_path` may point to the same file for
direct image fetch, but both fields must be explicit. This makes missing image
bytes distinct from missing source evidence.

Rules:

- Headers are allow-listed. Do not store `Set-Cookie`, `Cookie`,

- `primary_url_redacted` and `final_url_redacted` must be redacted before
  serialization.
- License hints are evidence only, not permission.

### Content Summary Schema

`content-summary.json` describes retrieved content without replacing image
acceptance:

```rust
pub struct ContentSummary {
    pub retrieval_job_id: RetrievalJobId,
    pub candidate_id: CandidateId,
    pub content_kind: RetrievedContentKind,
    pub summary_text: String,
    pub summary_quality: SummaryQuality,
    pub summary_quality_gate_passed: bool,

```

Minimum quality gate:

- Summary is non-empty.
- It states that an image artifact was fetched or explains why not.
- It does not claim semantic relevance approval.
- It references artifact, sidecar, and task report paths.
- It is marked generated by local retrieval adapter, self-hosted service, paid
  service, or fixture.

## Channel-Specific Design

### Normal Web Fetch

Normal web fetch is the default free local path. It includes direct image fetch
and source-page resolve as separate attempt modes.

Direct image fetch:

1. Evaluate pre-fetch policy for URL, domain, robots setting, login/paywall
   hints, prohibited sources, and paid status.
2. HTTP GET `primary_image_url` without credentials, cookies, login flows, or
   access bypass.
3. Reject 401/403 as access restricted and non-fallbackable.
4. Reject HTML, JSON, metadata, empty bodies, or source pages as
   `MetadataOnlyRejected` unless source-page resolve is the explicit next
   attempt.
5. Write local/source artifact.
6. Compute SHA-256 checksum.
7. Reconcile reported content type and sniffed content type.
8. Probe dimensions when supported.
9. Generate sidecar, content summary, task report, visual description,
   diagnostics, and fetch trace.

Source-page resolve:

1. Use `alternate_source_page_url` only when direct fetch failed with a
   fallbackable reason or direct URL is missing but source page is allowed.
2. Fetch source page without credentials and without executing login bypass.
3. Extract candidate image URLs using safe static parsing: OpenGraph image,
   Twitter image, schema.org image, direct image links, or canonical image
   elements.
4. Choose only URLs that remain within policy boundaries.
5. Fetch the selected image and produce the same artifact set.
6. Record both source-page and image-fetch attempts.

Normal web fetch must not execute arbitrary JavaScript in v1.1 unless a future
policy and dependency decision explicitly adds a local rendering channel.

### Self-Hosted Service

Self-hosted retrieval is the open-source service fallback boundary. It is
disabled until configured.

Adapter contract:

- Readiness checks endpoint, dependency, credential env var if required,
  health probe, and capability support.
- Requests must include `RetrievalJob`, redacted URLs, requested outputs, and
  policy context.
- Responses must be converted into local artifact files under the run staging
  directory.
- If service returns its own sidecar/summary/report/description, adapter must
  validate schema and copy them into local staging.
- If service returns bytes but not evidence, adapter may synthesize missing
  evidence locally.
- If service returns only remote URLs or metadata, result is
  `MetadataOnlyRejected`.

Self-hosted service cannot bypass access restrictions discovered by normal web
fetch. If the previous failure was 401/403/login/paywall/prohibited source,
fallback to self-hosted is stopped unless product policy explicitly permits a
different non-bypass interpretation.

### Paid Online Service

Paid retrieval is a configured but disabled-by-default fallback boundary.

Paid channel execution requires all of:

- `RuntimeConfig.policy.allow_paid_channels = true`.
- QueryPlan `retrieval_policy.allow_paid = true`.
- The paid channel config `enabled = true`.
- Budget or usage boundary configured.
- Credential env var present when required.
- Self-check reports paid readiness as available.

If any condition is absent:

- Readiness is `RETRIEVAL_PAID_UNCONFIRMED`.
- Attempt is not executed.
- Batch result records `PaidUnconfirmed` and a blocker diagnostic.

Paid service responses must satisfy the same artifact contract as other
channels. Paid output cannot be accepted if it is metadata-only or remote-only.

## Control Flow

Retrieval begins only after TASK-003 candidate quality has produced a
`RetrievableCandidateBatch`.

```text
NormalizedQueryPlan + RuntimeConfig.retrieval_channels
  + RetrievableCandidateBatch from TASK-003
  -> evaluate retrieval channel readiness
  -> plan RetrievalBatch target = required_image_count * 2
  -> create RetrievalJob per selected retrievable candidate
  -> for each job, attempt normal_web_fetch.direct_image_fetch
  -> if fallbackable, attempt normal_web_fetch.source_page_resolve
  -> if fallbackable and policy allows, attempt self_hosted_service
  -> if fallbackable and paid explicitly allowed, attempt paid_online_service
  -> write artifacts and evidence
  -> validate artifact completeness
  -> produce RetrievalBatchResult
  -> TASK-003 image mechanical acceptance
  -> TASK-005 package materialization and validation
```

Detailed sequence:

1. Orchestrator passes `NormalizedQueryPlan`, `RuntimeConfig`, attempt counters,
   and `RetrievableCandidateBatch` to retrieval.
2. Retrieval evaluates configured channel readiness and policy.
3. If no normal web fetch channel is usable, retrieval returns an execution
   blocker. It may still report self-hosted/paid readiness, but cannot silently
   start with paid.
4. Batch planner selects up to `retrieval_batch_target` candidates by priority.
5. Planner emits `RetrievalBatchShortage` when selected candidates are fewer
   than target.
6. For each job, executor performs attempts in fallback order.
7. Every attempt records `RetrievalAttemptTrace`.
8. The first complete artifact result terminates that job.
9. Non-fallbackable blockers terminate that job and prevent higher-tier bypass.
10. After all jobs terminate, retrieval returns `RetrievalBatchResult`.

Retrieval can return a mix of complete, partial, failed, and blocked results.
The orchestrator does not count complete retrieval results as accepted images
until TASK-003 image acceptance passes.

## State Persistence

Retrieval owns staging state during a run. TASK-005 owns final package
materialization.

Recommended in-memory state:

```rust

    pub full_attempt_count: u8,
    pub retry_count: u8,
    pub retrieval_batch_id: String,
    pub target_size: u32,
    pub active_jobs: BTreeMap<RetrievalJobId, RetrievalJob>,
    pub terminal_results: BTreeMap<RetrievalJobId, RetrievalArtifactResult>,
    pub attempt_trace: Vec<RetrievalAttemptTrace>,
    pub fallback_decisions: Vec<RetrievalFallbackDecision>,
}
```

Persistence rules:

- Write artifacts only under the configured run staging/output directory.
- Use atomic write patterns where practical: write to temporary file, fsync or
  flush when supported, then rename into final artifact path.
- Do not delete partial artifacts during the attempt; keep them as diagnostics
  when package policy allows.
- Do not store resolved credentials in state, trace, sidecar, summary, task
  report, or diagnostics.
- Path values in DTOs should be relative to run/package root when serialized by
  TASK-005, while in-memory paths may be absolute `PathBuf`s.

State invariants:

- `retry_count = full_attempt_count - 1`.
- `retrieval_batch_id` is unique per full attempt.
- Each `retrieval_job_id` appears in exactly one batch.
- Every terminal result belongs to one known job.
- Every artifact path in a complete result exists at the time retrieval returns.
- Metadata-only results cannot have `RetrievalStatus::Complete`.

## Security Permissions And Policy

Security rules:

- Retrieval channels may resolve credential env vars internally but must not
  serialize resolved values.
- HTTP requests must not send cookies, authorization headers, login credentials,
  or bypass tokens unless a future explicit policy adds that capability.
- URLs are redacted before sidecar, summary, task report, trace, diagnostics,
  logs, and package output.
- Signed URLs or token-bearing query parameters must be stripped or replaced
  with redacted markers.
- Paid channel cannot run unless both runtime config and QueryPlan policy allow
  it and budget/confirmation is present.
- Access-restricted results cannot be escalated to self-hosted or paid as a
  bypass.
- Prohibited domains are blocked before network fetch.
- Unknown authorization risk is retained as evidence; configured policy decides
  warning vs blocker.
- Fixture channel output is test-only and must not count as production
  retrieval evidence.

Robots/site-rule behavior:

- If `respect_robots = true`, pre-fetch policy checks the configured
  robots/site-rule posture when available.
- If robots status is unknown and policy is `warn`, record
  `RETRIEVAL_ROBOTS_UNDECIDED` as warning and continue.
- If robots status is unknown and policy is `block`, do not fetch and return
  `PolicyBlocked`.
- If robots explicitly disallow fetch, return `RETRIEVAL_ROBOTS_BLOCKED`.

## Delivery Qualification Boundary

Retrieval never decides delivered images.

Allowed retrieval conclusions:

- Artifact retrieval is complete.
- Artifact retrieval is partial.
- Artifact retrieval failed.
- Artifact retrieval is blocked by policy, access, paid, or readiness.
- Artifact evidence is invalid or metadata-only.

Forbidden retrieval conclusions:

- The image is accepted.
- The image is delivered.
- The image semantically satisfies the QueryPlan.
- Qwen 3.5 VLM approved the image.

Only TASK-003 image mechanical acceptance plus Qwen 3.5 VLM image evaluation can
produce `DeliveredQualified`, and TASK-005 package validation still verifies
that every delivered image has complete artifact evidence.
