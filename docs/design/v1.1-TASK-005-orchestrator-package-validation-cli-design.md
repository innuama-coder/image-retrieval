This document defines the v1.1 detailed design for the complete run loop,
attempt and retry state, canonical delivery package, deterministic package
validation, review and handoff files, CLI commands, self-check behavior,
diagnostics, observability, and release-blocking external dependency states.

This is documentation-only detailed design. It does not implement Rust source,
tests, Cargo manifest changes, provider adapters, retrieval channels, or
Qwen 3.5 VLM adapter code.

## Scope And Boundary

### In Scope

- Complete `run` orchestration from admission through search, candidate
  quality, retrieval, image acceptance, package build, validation, review, and
  handoff.
- Runtime state machine and attempt accounting with distinct
  `full_attempt_count` and `retry_count`.
- Rules for 1 initial full attempt plus up to 3 retries, accepted-image carry
  forward, retry exhaustion, `passed`, `partial`, and `blocked` package status.
- Canonical package root and schemas for:
  `image-recalls.json`, `retrieved-images.json`, `coverage-report.json`,
  `retrieval-manifest.json`, `package-summary.json`,
  `delivery-report.json`, `validation.json`, `review.json`, and
  `handoff-report.json`.
- Package validation contract, validation failure codes, and CLI
  `validate-package` behavior.
- CLI command surface for `run`, `self-check`, `validate-package`, and
  `inspect-package`.
- Self-check output contract that reports search provider, retrieval channel,
  Qwen 3.5 VLM, policy, credential, output, and validator readiness.
- Security redaction, package-safe diagnostics, audit metrics, trace links,
  exit codes, and development verification plan.

### Forbidden Scope And Non-Goals

- Do not implement code in this task.
- Do not modify source, tests, Cargo manifests, or other design documents.
- Do not choose the default broad image search provider.
- Do not implement the production Qwen 3.5 VLM adapter beyond consuming TASK-003
  readiness and decisions.
- Do not add provider-specific API schemas or retrieval channel internals owned
  by TASK-002 and TASK-004.
- Do not let fixture, metadata-only, thumbnail-only, source-page-only, or
  summary-only evidence count as production delivery.
- Do not silently enable paid retrieval or bypass robots, authorization,
  login, or paywall policy.

## Current Source Delta

The current codebase contains useful anchors:

- `src/orchestrator/mod.rs` has `OrchestratorState`, `AttemptCounter`, and
  delivery decision construction.
- `AttemptCounter::new` starts at `full_attempt_count = 1` and
  `retry_count = 0`, matching the v1.1 invariant.
- `process_image_acceptance` already decides between full delivery, retry, and
  limited delivery based on accepted images and retry exhaustion.
- `src/delivery/mod.rs` has a package writer, directory creation, image copy,
  evidence and diagnostics writing, and serialization patterns.
- `src/main.rs` has Clap-based subcommands and self-check config loading.

The v1.1 gaps are structural:

- `run` stops after QueryPlan validation and `TaskPlan` summary; it does not
  call search, candidate quality, retrieval, image acceptance, delivery, or
  validation.
- `TaskOrchestrator` only processes image acceptance results and does not own a
  full pipeline attempt.
- Current delivery package uses `status.json`, `manifest.json`, and
  `summary.md`; v1.1 requires the canonical package files listed above.
- There is no independent `PackageValidator` module or CLI
  `validate-package` command.
- `self-check` uses MVP provider/channel readiness entries and infers
  Qwen 3.5 VLM readiness from provider readiness; v1.1 needs explicit readiness
  sections from TASK-001 through TASK-004.
- Existing user-facing strings include encoding corruption; v1.1 CLI output
  should use valid UTF-8 and machine-readable JSON for automation.

## Interfaces Types And DTO Contracts

### Run Orchestrator Trait Boundary

Development may keep a concrete `TaskOrchestrator`, but v1.1 should expose a
single full-workflow entry point:

```rust
pub struct RunRequest {
    pub query_plan: NormalizedQueryPlan,
    pub runtime_config: RuntimeConfig,
    pub output_dir: PathBuf,
    pub execution_mode: ExecutionMode,
    pub run_id: String,
}

pub struct RunOutcome {
    pub run_id: String,
    pub query_plan_id: QueryPlanId,
    pub status: PackageStatus,
    pub full_attempt_count: u8,
    pub retry_count: u8,

    pub validation_status: ValidationStatus,

```

`ExecutionMode` values:

- `Production`: fixture evidence cannot pass readiness or delivery.
- `Fixture`: deterministic tests may run; package status must show fixture mode.
- `DryRun`: admission and readiness only; no search or retrieval calls.

### Run State

```rust
pub struct RunState {
    pub run_id: String,
    pub query_plan_id: QueryPlanId,
    pub status: PackageStatus,
    pub full_attempt_count: u8,
    pub retry_count: u8,
    pub retry_limit: u8,
    pub full_attempt_limit: u8,
    pub required_image_count: u32,
    pub accepted_images: Vec<DeliveredImageRecord>,
    pub rejected_candidates: Vec<CandidateQualityDecision>,
    pub retrieval_failures: Vec<RetrievalArtifactResult>,
    pub gaps: Vec<CoverageGap>,
    pub attempts: Vec<RunAttemptRecord>,
    pub diagnostics: Vec<WorkflowDiagnostic>,
}
```

Invariants:

- Initial state has `full_attempt_count = 1` and `retry_count = 0`.
- `retry_count = full_attempt_count - 1` always.
- `retry_limit = 3` unless a stricter admitted QueryPlan value is supplied.
- `full_attempt_limit = 1 + retry_limit`; default is `4`.
- `accepted_images.len() <= required_image_count` in final delivered output.
- Accepted images are carried across attempts and not re-retrieved.
- A package cannot be `passed` unless `accepted_image_count >= required_image_count`
  and deterministic validation passes.

```rust
    pub finished_at: Option<OffsetDateTime>,

    pub image_acceptance_outcome: Option<ImageAcceptanceOutcome>,

    pub gap_delta: Vec<CoverageGap>,
    pub terminal_reason: Option<WorkflowFailureCode>,

```

The package builder may serialize full attempt records directly or through
redacted projections. Raw secrets and authenticated URLs are forbidden.

### Package Status

```rust
pub enum PackageStatus {
    Passed,
    Partial,
    Blocked,
}
```

Status mapping:

- `Passed`: requested count reached and package validation passes.
- `Partial`: at least one accepted image exists, requested count remains unmet,
  retry limit is exhausted, and package validation passes for the limited
  delivery package.
- `Blocked`: no accepted artifact-backed image exists, or a hard dependency,
  policy, input, package validation, or validation command prevents honest
  delivery.

Input rejection does not produce a delivery package unless the user explicitly
requests a diagnostics package. Normal `run` exits before package build on
admission rejection.

## Control Flow

### Full Run Loop

```text
load query plan
load runtime config
admit and normalize query plan
build readiness reports
if admission or hard readiness blocks:
  build blocked diagnostics package if configured
  exit blocked

state.full_attempt_count = 1
state.retry_count = 0
state.retry_limit = normalized.retry_limit
state.full_attempt_limit = 1 + retry_limit

while state.full_attempt_count <= state.full_attempt_limit:
  attempt = start attempt record

  search_outcome = search(normalized, config, attempt counters)
  candidate_quality = evaluate_candidates(search_outcome)
  if candidate quality has execution block and no retrievable candidates:
    record gap and attempt blocker

  retrieval_batch = plan batch(candidate_quality.retrievable_batch)
  retrieval_result = execute retrieval(retrieval_batch)
  image_acceptance = evaluate_images(retrieval_result)

  append accepted images not already accepted
  append rejected candidates, retrieval failures, diagnostics, gaps

  if accepted_count >= required_image_count:
    state.status = Passed
    break

  if state.retry_count < state.retry_limit:
    state.retry_count += 1
    state.full_attempt_count += 1
    expand search or fallback context, excluding accepted images
    continue

  state.status = Partial if accepted_count > 0 else Blocked
  break

package_dir = build canonical package(state)
validation = validate package(package_dir)
review = build review(state, validation)
handoff = build handoff(state, validation, review)
rewrite validation/review/handoff files if needed
return RunOutcome
```

### State Machine

```text
admission
  -> readiness_check
  -> attempt_started
  -> searching
  -> candidate_quality
  -> retrieval_planning
  -> retrieving
  -> image_acceptance
  -> coverage_check
  -> retrying
  -> attempt_started
  -> package_build
  -> package_validation
  -> review
  -> handoff
  -> passed | partial | blocked
```

Hard transitions:

- Admission error -> CLI exit `InputError`; no normal delivery package.
- No available search provider -> `Blocked`.
- Production Qwen 3.5 VLM unavailable when subjective evaluation is required ->
  attempt execution block; retry may run only if configuration changes are not
  required. In normal single-run execution this usually ends as `Blocked`.
- Retrieval policy hard block for every candidate -> retry if candidates or
  policy alternatives exist, else `Blocked`.
- Accepted image count reached -> `PackageBuild`.
- Retry exhausted with accepted images -> `Partial`.
- Retry exhausted with zero accepted images -> `Blocked`.
- Package validation fail after build -> final status cannot be `Passed`.

### Retry Semantics

- `full_attempt_count` counts the initial attempt and all retries.
- `retry_count` counts only attempts after the initial attempt.
- Default sequence:
  - Attempt 1: `full_attempt_count = 1`, `retry_count = 0`
  - Attempt 2: `full_attempt_count = 2`, `retry_count = 1`
  - Attempt 3: `full_attempt_count = 3`, `retry_count = 2`
  - Attempt 4: `full_attempt_count = 4`, `retry_count = 3`
- No attempt may start with `retry_count > 3`.
- Validation fails any package where
  `retry_count != full_attempt_count - 1`.
- Validation fails any package where `full_attempt_count > 4` unless a future
  product version explicitly changes the constitution.

`coverage-report.json` aggregates them into final reasons.

## Data Flow

```text
NormalizedQueryPlan + RuntimeConfig
  -> RunRequest
  -> provider/channel/Qwen 3.5 VLM/policy readiness reports
  -> SearchSessionOutcome
  -> CandidateQualityOutcome
  -> RetrievableCandidateBatch
  -> RetrievalBatch
  -> RetrievalBatchResult
  -> ImageAcceptanceOutcome
  -> RunState
  -> CanonicalPackageInputs
  -> package files
  -> PackageValidationReport
  -> review and handoff
```

Package-safe input records:

- Redacted `NormalizedQueryPlan`.
- Redacted config summary and readiness reports.
- Search recalls and provider usage events from TASK-002.
- Candidate and image quality decisions from TASK-003.
- Retrieval artifacts and attempt trace from TASK-004.
- Orchestrator attempt records and gap accounting from this design.

Package-forbidden values:

- Resolved credentials, tokens, cookies, authorization headers, private keys.
- Full authenticated URLs or signed URLs with secret query parameters.
- Raw provider responses that have not been allow-listed and redacted.
- Raw Qwen 3.5 VLM transcripts beyond approved rationale summaries.

```json
  "blockers": [],
  "safe_for_downstream": true
}
```

Handoff statuses:

- `ready`
- `limited`
- `blocked`

`partial` packages usually map to `limited`. `blocked` packages map to
`blocked`.


Build order:

1. Create package root and subdirectories.
2. Copy accepted image artifacts into `images/`.
3. Copy or synthesize package-safe evidence into `evidence/`.
4. Write diagnostics into `diagnostics/`.
5. Write `image-recalls.json`.
6. Write `retrieved-images.json`.
7. Write `coverage-report.json`.
8. Write `retrieval-manifest.json`.
9. Write `package-summary.json`.
10. Write provisional `delivery-report.json`.
11. Run validator and write `validation.json`.
12. Write `review.json`.
13. Write `handoff-report.json`.

Atomicity rules:

- Each JSON file should be written to a temporary path then renamed.
- Package validation must run after all required files exist.
- If validation fails, keep the package but set final status to `blocked` or
  `partial` according to validation result and accepted evidence.
- Do not delete failed or rejected evidence unless it contains unrecoverable
  sensitive data; in that case replace with redacted diagnostic evidence and
  block validation.

### Validation Fail Rules

Validation fails when any of the following is true:

- Required canonical package file is missing.
- Required JSON file is unreadable or invalid.
- Required package file has a mismatched `query_plan_id`.
- `package-summary.json.status = "passed"` but validation status is not pass.
- Delivered image lacks `retrieval_job_id`.
- Delivered image lacks `candidate_id` or `query_plan_id`.
- Delivered image lacks local artifact, source artifact, sidecar, summary,
  task report, visual description, checksum, content type, or media match.
- Required artifact path does not exist.
- A delivered image artifact is empty.
- A delivered image is metadata-only, thumbnail-only, source-page-only,
  sidecar-only, summary-only, or task-report-only.
- Candidate ownership does not match QueryPlan.
- Retrieval job ownership does not match candidate and QueryPlan.
- Candidate Qwen 3.5 VLM decision is missing for a delivered image.
- Image Qwen 3.5 VLM decision is missing for a delivered image.
- Mechanical candidate or image acceptance failed but item is delivered.
- Coverage counts do not match delivered images.
- `retry_count != full_attempt_count - 1`.
- `full_attempt_count > retry_limit + 1`.
- Default v1.1 package has `retry_count > 3`.
- A package marked `partial` has zero accepted images.
- A package marked `blocked` has accepted images but no validation explanation.
- Manifest links do not resolve.
- Provider readiness evidence is missing.
- Retrieval attempt trace is missing for a retrieval result.
- Sensitive credentials, tokens, cookies, authorization headers, or private
  keys appear in package files.
- Fixture evidence is used to mark production package as `passed`.

## Security Authorization And Redaction

Security rules:

- CLI must not print credentials, tokens, cookies, authorization headers, or
  private keys.
- Runtime config stores env var names, not resolved values, in package files.
- All package files and diagnostics are passed through central redaction or a
  package-safe serializer.
- Validator performs a sensitive-data scan across JSON, markdown, text, sidecar,
  summary, task report, trace, review, handoff, and diagnostics files.
- Absolute local paths may be used in runtime memory, but package JSON should
  prefer paths relative to package root.
- If an artifact or evidence file contains unredactable secrets, the builder
  must exclude it from delivery and record `PKG_SECRET_LEAK` or
  `WORKFLOW_SECRET_REDACTED`.
- Authorization and robots policy outcomes from TASK-001 and TASK-004 must be
  visible in `delivery-report.json`, `coverage-report.json`, and
  `handoff-report.json`.
- Paid channel usage must be visible in retrieval trace and package summary.

## Review And Handoff

Review is not a substitute for deterministic validation. For v1.1:

1. Deterministic package validation decides whether package structure and
   artifact evidence are valid.
2. Qwen 3.5 VLM candidate and image decisions decide subjective acceptance for
   individual items.
3. `review.json` aggregates the subjective and deterministic evidence into a
   downstream-readable verdict.
4. `handoff-report.json` states whether downstream consumers can use the
   package and what gaps remain.

Handoff rules:

- `Passed` plus validation pass plus review pass -> `handoff_status = ready`.
- `Partial` plus validation pass -> `handoff_status = limited`.
- `Blocked` or validation fail -> `handoff_status = blocked`.
- Handoff report must include unresolved release blockers even when the package
  contains useful limited artifacts.

## Verification Plan

### Unit Tests

- `AttemptCounter` starts with `full_attempt_count = 1` and `retry_count = 0`.
- `retry_count = full_attempt_count - 1` invariant is enforced.
- Default retry limit allows exactly four full attempts.
- `RunState` maps accepted count reaching required count to `Passed`.
- Retry exhaustion with accepted images maps to `Partial`.
- Retry exhaustion with zero accepted images maps to `Blocked`.
- `CoverageGap` computes missing count and gap type correctly.
- `CanonicalPackageBuilder` requires all canonical file names.
- Delivered image DTO rejects missing retrieval job id.
- `PackageValidator` fails missing required file.
- `PackageValidator` fails invalid JSON.
- `PackageValidator` fails missing artifact path.
- `PackageValidator` fails missing checksum.
- `PackageValidator` fails metadata-only delivered item.
- `PackageValidator` fails retry counter mismatch.
- `PackageValidator` fails secret-like package content.
- CLI exit code mapping is deterministic for passed, partial, blocked, input
  error, config error, and validation fail.

### Integration Tests

- `run` invokes admission, search, candidate quality, retrieval, image
  acceptance, package builder, validator, review, and handoff in order.
- Search outcome from TASK-002 appears in `image-recalls.json` and manifest.
- Candidate quality outcome from TASK-003 appears in manifest, review, and
  delivery report.
- Retrieval artifact result from TASK-004 appears in `retrieved-images.json`
  and manifest.
- Image acceptance outcome controls `delivered_images`.
- Partial delivery after four full attempts writes package with exact gaps.
- Blocked delivery with no accepted images writes blocked evidence when package
  generation is configured for blocked diagnostics.
- `validate-package` reads an existing package and reproduces validation
  verdict.
- `inspect-package` is read-only and can display summary, coverage, validation,
  handoff, and manifest sections.
- `self-check` reports separate provider, retrieval, Qwen 3.5 VLM, policy, output,
  credential, and validator readiness.

### Fixture E2E Tests

- Fixture mode can produce a deterministic `passed` fixture package marked as
  fixture mode.
- Production mode using fixture provider, fixture retrieval, or fixture
  Qwen 3.5 VLM cannot produce `Passed`.
- Missing sidecar in fixture package fails validation.
- Missing task report in fixture package fails validation.
- Broken manifest link in fixture package fails validation.
