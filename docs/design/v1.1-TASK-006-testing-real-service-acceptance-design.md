## Test Taxonomy

### Local Deterministic Tests

Local deterministic tests are required for every development PR and must not
require network, credentials, or production Qwen 3.5 VLM.

| Class | Purpose | Expected command |
| --- | --- | --- |
| Unit | Validate pure functions, DTO invariants, parsing, redaction, validation issue mapping, and package schema helpers. | `cargo test --all` |
| Integration | Validate module boundaries and handoff contracts across domain/search/quality/retrieval/orchestrator/delivery/validation. | `cargo test --all` |
| Fixture E2E | Exercise full pipeline under explicit fixture mode, including both success and negative paths. | `cargo test --all` |
| CLI golden | Run compiled CLI against fixture QueryPlans/configs and compare JSON/human output and exit codes. | `cargo test --all` or an explicit integration harness |
| Security regression | Scan generated package, logs, diagnostics, sidecars, summaries, task reports, and traces for secret-like values. | `cargo test --all` |

Fixture mode packages may be used to test structure and logic. They must carry
fixture evidence and cannot satisfy production real-service acceptance.

### Real-Service Smoke Tests

Real-service smoke tests are manually or CI-gated tests that require configured
external services and explicit product decisions. They are not part of the
default local `cargo test --all` suite unless the environment opt-in flag is
present.

The minimum smoke objective is:

1. A real configured broad search provider produces at least one normalized
   candidate.
2. An open-license or configured provider path can also be checked when the
   provider list gate is closed.
3. A non-fixture retrieval channel produces at least one complete
   `RetrievalArtifactResult`.
4. Production Qwen 3.5 VLM candidate evaluation and image evaluation both execute.
5. At least one image reaches `DeliveredQualified`.
6. The generated canonical package validates successfully.
7. The smoke report records exact provider/channel/Qwen 3.5 VLM/policy readiness
   and any gaps.

If prerequisites are absent, the test outcome is `blocked` or `skipped`, not
`passed`.

## Unit Test Design

### QueryPlan Config Policy

Required tests:

- `required_image_count` defaults to `1`.
- `quality` defaults to `general`.
- `required_count` input alias is accepted but canonical output uses
  `required_image_count`.
- `query_texts` defaults to `[description]`.
- Explicit zero count defaults to one with warning.
- `candidate_target = required_image_count * 20`.
- `retrieval_batch_target = required_image_count * 2`.
- `retry_limit` defaults to `3`; values above `3` reject.
- `full_attempt_limit = 1 + retry_limit`.
- `retry_count = full_attempt_count - 1` invariant is enforced.
- Config stores credential env var names, never resolved values.
- Paid retrieval defaults disabled.
- Fixture provider/channel/Qwen 3.5 VLM readiness cannot satisfy production mode.
- Redaction removes API keys, bearer tokens, cookies, passwords, and signed URL
  query values from diagnostics.

### Provider Contract

Each provider adapter must pass the same contract tests:

- Implements object-safe `BaseSearchProvider`.
- `readiness(config)` returns structured status without full search execution.
- Missing credential env var maps to `PROVIDER_CREDENTIAL_MISSING`.
- Disabled provider maps to `PROVIDER_DISABLED`.
- Health failure maps to `PROVIDER_HEALTH_FAILED`.
- Quota or rate-limit evidence maps to provider failure codes.
- Search request includes no credential values.
- Provider response normalization rejects missing image URL.
- Candidate records include query ownership, provider id, provider rank,
  search round, image URL, source page URL when available, MIME/dimension
  evidence, license hint, dedupe key, origin IDs, and provenance.
- Raw provider response blobs are not serialized into package-safe DTOs.
- Fixture providers are marked fixture-only and blocked in production mode.

### Candidate And Image Quality

Required tests:

- Candidate missing/invalid image URL is mechanically blocked.
- Candidate ownership mismatch is blocked.
- Prohibited source and access-restricted policy are blocked.
- Negative-scope contradiction is blocked.
- Missing source page, license, or dimensions produce reference metrics unless
  policy promotes them to blockers.
- Mechanically blocked candidates are not submitted to Qwen 3.5 VLM.
- Qwen 3.5 VLM approve/reject/uncertain/unexecutable map to retrievable/rejected/
  uncertain/execution-blocked statuses.
- Qwen 3.5 VLM response with missing or unknown subject id is invalid.
- Production Qwen 3.5 VLM unavailable blocks subjective evaluation.
- Fixture Qwen 3.5 VLM in production maps to fixture-not-production blocker.
- Retrieved image missing local artifact, source artifact, sidecar, summary,
  task report, visual description, checksum, media match, or ownership is
  mechanically blocked.
- Metadata-only and thumbnail-only retrieval results are mechanically blocked.
- Image delivery requires both mechanical pass and Qwen 3.5 VLM image approval.

### Retrieval Artifact Channels

Required tests:

- `RetrievalBatch` uses `retrieval_batch_target`, not an ad hoc count.
- `RetrievalJob` ids are unique and preserve query/candidate ownership.
- Short batch evidence includes target, actual, shortage code, and candidate
  quality context.
- Channel readiness reports disabled, dependency missing, misconfigured,
  missing credential, paid unconfirmed, robots undecided, and fixture-only
  states.
- Direct image fetch complete result writes every required artifact and trace.
- Source-page resolve is a separate attempt mode and records both page and
  artifact attempts.
- Access restriction does not fallback to self-hosted or paid as a bypass.
- Paid channel is not invoked without runtime and QueryPlan approval.
- Self-hosted and paid service responses that contain only remote URLs or
  metadata map to `MetadataOnlyRejected`.
- Missing sidecar, summary, task report, visual description, checksum, or local
  artifact prevents `RetrievalStatus::Complete`.
- Token-bearing URLs are redacted in sidecar, summary, task report, diagnostics,
  and trace.

### Orchestrator Delivery Validation

Required tests:

- `run` stage order is deterministic and auditable.
- Accepted images are carried across retries and not re-retrieved.
- Previously failed candidates are not retried through the same failed channel
  unless policy allows a different fallback path.
- Requested count reached maps to `passed` only after package validation pass.
- Retry exhausted with accepted images maps to `partial`.
- Retry exhausted with zero accepted images maps to `blocked`.
- Package validation failure prevents final `passed`.
- `CoverageGap` aggregation records missing count, source stage, failure code,
  retryability, and evidence refs.
- Review and handoff statuses derive from package status plus validation
  verdict.

## Integration And Fixture E2E Design

### Integration Suites

| Suite | Required scenarios |
| --- | --- |
| `query_config_integration` | QueryPlan plus RuntimeConfig admission; policy broadening blocked; redacted config summary; self-check readiness input. |
| `search_provider_integration` | Registry from config, provider readiness, weighted scheduling, dedupe provenance, no providers blocked, package-safe `image-recalls.json`. |
| `quality_vlm_integration` | Search outcome to candidate quality, Qwen 3.5 VLM unavailable, retrievable batch, retrieval result to image acceptance, missing artifact rejected before Qwen 3.5 VLM. |
| `retrieval_artifact_integration` | Batch planning, direct fetch artifact set, source-page fallback, self-hosted disabled, paid disabled, metadata-only rejected, trace emitted. |
| `orchestrator_package_integration` | Full pipeline with fixture adapters, retry exhaustion, partial/blocked package, canonical files, validation/review/handoff. |
| `validation_integration` | `validate-package` reproduces report; broken package fixtures fail with expected issue codes. |

### Fixture E2E Scenarios

Fixture E2E tests must be explicit about `ExecutionMode::Fixture` and must
produce package evidence marked as fixture mode.

Required fixture scenarios:

- `fixture_passed_package`: full pipeline reaches `passed` with canonical files.
- `fixture_partial_package`: one accepted image with requested count unmet after
  four full attempts.
- `fixture_blocked_no_provider`: no ready production-capable provider.
- `fixture_blocked_vlm`: production Qwen 3.5 VLM unavailable.
- `fixture_blocked_retrieval_artifact_missing`: retrieval missing sidecar,
  summary, task report, visual description, or checksum.
- `fixture_metadata_only_rejected`: metadata/page/summary-only result never
  appears in delivered images.
- `fixture_access_restricted_no_fallback`: access restriction stops fallback.
- `fixture_paid_unconfirmed`: paid channel is abandoned or blocked.
- `fixture_secret_leak_validation_fail`: validator finds a seeded fake secret.
- `fixture_production_mode_rejects_fixture_evidence`: production run with any
  fixture provider/channel/Qwen 3.5 VLM cannot produce `passed`.

## CLI Golden Test Design

CLI tests should use a compiled binary through Cargo integration tests or a
stable command harness. Golden files must avoid absolute temp paths when
possible; paths should be normalized before comparison.

Required commands:

```bash
image-retrieval run --query-plan tests/fixtures/v1_1/query-plan-basic.json --config tests/fixtures/v1_1/config-fixture.toml --output-dir <tmp> --mode fixture --format json
image-retrieval self-check --config tests/fixtures/v1_1/config-fixture.toml --query-plan tests/fixtures/v1_1/query-plan-basic.json --format json
image-retrieval validate-package --package-dir <tmp>/package --format json
image-retrieval inspect-package --package-dir <tmp>/package --show all --format json
```

Golden assertions:

- `run` in fixture mode returns exit code `0` only for fixture `passed` package.
- Production mode with fixture evidence returns `DeliveryBlocked` or
  `ReadinessBlocked`, not success.
- `run` partial returns `PartialDelivery` unless `--fail-on-partial` maps it to
  `DeliveryBlocked`.
- `validate-package` returns `PackageValidationFailed` for broken package
  fixtures.
- `inspect-package` does not modify file mtimes or package content.
- Human output contains package path, status, accepted count, gap count, and
  primary blocker; JSON output contains the same fields in machine-readable
  form.
- CLI output does not include credentials, tokens, cookies, authorization
  headers, or raw secret-like values.

## Real-Service Smoke Prerequisites

Real-service smoke is eligible only when all prerequisite gates below are
closed or explicitly waived in a signed acceptance note.

| Prerequisite | Required evidence |
| --- | --- |
| Default SerpApi provider readiness | `serpapi_google_images` adapter exists, config points to `https://serpapi.com/search` with `engine=google_images`, and `SERPAPI_API_KEY` or an explicit overriding credential env var is present. |
| Built-in provider policy | List of bundled/allowed/restricted providers and any opt-in requirements. |
| Real provider credentials | Environment variables present by name and readable by adapter. Values are never printed. |
| Qwen 3.5 VLM adapter readiness | `qwen_3_5_vlm` adapter exists, endpoint/base URL and model id are configured, `QWEN_API_TOKEN` or configured equivalent is present, candidate/image prompt templates exist, and readiness probe passes. |
| Artifact-capable retrieval channel | At least normal web fetch enabled and output directory writable; self-hosted/paid only if configured. |
| Paid retrieval policy | Explicit runtime config and QueryPlan allowance when paid channel is in scope. |
| robots/site-rule policy | `warn` or `block` behavior selected and visible in self-check. |
| Authorization blocking rules | Unknown/prohibited/access-restricted handling selected. |
| Quality calibration or waiver | Minimum dimensions/thresholds and waiver status recorded. |

Recommended environment variable names:

| Variable | Purpose |
| --- | --- |
| `IMAGE_RETRIEVAL_REAL_SMOKE=1` | Opt-in flag that allows real-service smoke to run. |
| `IMAGE_RETRIEVAL_CONFIG` | Path to production-like runtime config. |
| `IMAGE_RETRIEVAL_QUERY_PLAN` | Path to smoke QueryPlan. |
| `IMAGE_RETRIEVAL_OUTPUT_DIR` | Path to writable output directory. |
| `SERPAPI_API_KEY` | Default SerpApi Google Images credential env var. A config may override the env var name, but output must print only the env var name, never its value. |
| Provider-specific credential env vars | Names referenced by `RuntimeConfig.providers[*].credential_env`. |
| Retrieval-specific credential env vars | Names referenced by `RuntimeConfig.retrieval_channels[*].credential_env`. |
| Qwen 3.5 VLM credential env var | Name referenced by `RuntimeConfig.vlm_evaluation.credential_env`. |
| `IMAGE_RETRIEVAL_ALLOW_PAID=1` | Optional paid-channel smoke opt-in when the release gate allows it. |

The exact provider credential variable names are defined by config, not by this
document. Test output may print env var names but must never print resolved
values.

## Real-Service Smoke Flow

The smoke harness should run:

```bash
image-retrieval self-check --config "$IMAGE_RETRIEVAL_CONFIG" --query-plan "$IMAGE_RETRIEVAL_QUERY_PLAN" --format json
image-retrieval run --query-plan "$IMAGE_RETRIEVAL_QUERY_PLAN" --config "$IMAGE_RETRIEVAL_CONFIG" --output-dir "$IMAGE_RETRIEVAL_OUTPUT_DIR" --mode production --format json
image-retrieval validate-package --package-dir "$IMAGE_RETRIEVAL_OUTPUT_DIR/package" --format json
```

Acceptance assertions:

- `self-check.status` is `ready` or `warning`; no blocker for provider,
  retrieval, Qwen 3.5 VLM, policy, output, or validator readiness.
- At least one real provider readiness report is available and non-fixture.
- At least one search attempt returns normalized candidates.
- At least one non-fixture retrieval result has `retrieval_status = complete`.
- Complete retrieval results contain every required artifact field.
- Candidate and image Qwen 3.5 VLM decisions exist and are non-fixture.
- At least one delivered image is artifact-backed and accepted.
- Package status is `passed` for full smoke acceptance. A `partial` package can
  be recorded as useful evidence but cannot close v1.1 full acceptance unless
  product explicitly waives required count.
- `validate-package` passes.
- Secret scan passes.

```json
    }
  ],
  "notes": "No production Qwen 3.5 VLM adapter readiness was available."
}
```

Status values:

- `passed`: smoke ran and all acceptance assertions passed.
- `failed`: smoke ran but an assertion failed.
- `blocked`: smoke could not honestly run because a required dependency,
  credential, policy decision, or release gate is missing.
- `skipped`: smoke was intentionally not selected, usually because
  `IMAGE_RETRIEVAL_REAL_SMOKE` was not set.

Rules:

- A skipped smoke test cannot close real-service acceptance.
- A blocked smoke test is valid evidence for release gate tracking but not for
  v1.1 completion.
- A failed smoke test must preserve package and command evidence when safe.
- The report must list commands actually run and commands deliberately not run.
- The report must not contain resolved credential values.

## Acceptance Report Format

The final testing acceptance report should be written as
`docs/v1.1/TESTING-ACCEPTANCE.md` or an equivalent release artifact during the
testing/acceptance phase. TASK-006 only defines the format.

Required sections:

1. Scope and version under test.
2. Git commit, branch, and local dirty-state note.
3. Commands run, timestamps, exit codes, and summary output.
4. PRD AC matrix with pass/fail/blocked/skipped status.
5. Local deterministic test summary.
6. Fixture E2E summary.
7. Package validation summary.
8. CLI golden summary.
9. Security/redaction summary.
10. Real-service smoke summary.
11. Release gates and waivers.
12. Known residual risks.
13. Final verdict: `accepted`, `accepted_with_waiver`, or `rejected`.

Example row:

| AC | Status | Evidence | Notes |
| --- | --- | --- | --- |
| AC-008 | passed | `retrieval_artifact_direct_fetch_complete` and real smoke package `retrieved-images.json` | All required artifact fields present. |

Final verdict rules:

- `accepted` requires all AC rows passed and real-service smoke passed.
- `accepted_with_waiver` requires explicit product owner waivers for every
  blocked real-service or release gate item.
- `rejected` is required when any P0 AC fails without waiver.
