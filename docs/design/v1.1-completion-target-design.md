# image-retrieval v1.1 Completion Target Design

## Purpose

v1.1 is the first version that must satisfy the repository constitution as a
working Rust CLI for image search, retrieval, validation, and delivery
packaging.

This document is the design baseline for completing v1.1. It intentionally
does not import or reintroduce any gstack routing rules or workspace files. The
external image-search-retrieval workflow is used only as a capability reference:
multi-provider recall, auditable artifact retrieval, validation, review, and
honest package status.

## v1.1 Product Definition

Given a `QueryPlan`, the CLI must:

1. Validate and normalize the query plan.
2. Search image candidates through configured pluggable search providers.
3. Mechanically validate and subjectively rank candidates.
4. Retrieve selected candidates in batches through configured retrieval
   channels with fallback.
5. Mechanically validate and subjectively evaluate retrieved images.
6. Repeat the full workflow until the requested count is reached or retry
   limits are exhausted.
7. Produce an auditable delivery package with accepted images, gaps, evidence,
   validation, review, and handoff status.

The CLI is complete for v1.1 only when a real configured run can produce local
image artifacts, not just candidate metadata or image URLs.

## Non-Goals

- No web app, SaaS service, multi-user service, or account system.
- No image generation, editing, cropping, enhancement, or upscaling.
- No fixture or mock result may count as production delivery evidence.
- No hard-coded credentials.
- No silent use of paid retrieval channels.

## Core Defaults

| Field | Default / Rule |
|---|---|
| `required_image_count` | `1` when omitted |
| `quality` | `general` when omitted |
| Search candidate target | `required_image_count * 20` |
| Retrieval batch target per attempt | `required_image_count * 2` |
| `retry_count` limit | `3` |
| `full_attempt_count` limit | `1 initial attempt + 3 retries = 4` |
| Limited delivery | Allowed only after retry limit is exhausted |

`retry_count` and `full_attempt_count` must remain distinct in code, package
metadata, logs, and user-facing summaries.

## End-to-End Pipeline

```text
admission
  -> provider-readiness
  -> recall-planning
  -> multi-provider-search
  -> candidate-normalization
  -> candidate-quality
  -> retrieval-batch-planning
  -> retrieval-channel-execution
  -> image-acceptance
  -> coverage-accounting
  -> package-materialization
  -> validation
  -> review
  -> handoff
```

Hard gates:

- Admission failure stops before provider calls.
- No usable search provider blocks the run.
- Production Qwen 3.5 VLM unavailability blocks candidate or image subjective
  evaluation; fixture evaluation cannot substitute.
- No accepted image artifacts after all attempts produces `limited_delivery`
  or `blocked`, never `full_delivery`.
- Metadata-only retrieval is always rejected.

## QueryPlan v1.1

The core of `QueryPlan` remains the semantic image description. v1.1 also needs
structured execution requirements:

```json
{
  "description": "Everest South Col route map",
  "required_image_count": 2,
  "quality": "general",
  "query_texts": ["Everest South Col route map"],
  "material_types": ["map", "diagram"],
  "visual_requirements": ["route line visible"],
  "negative_scope": ["summit tourist photo without route context"],
  "source_diversity_requirement": 2,
  "provider_policy": {
    "provider_order": ["serpapi_google_images", "commons"],
    "cost_mode": "balanced"
  },
  "retrieval_policy": {
    "allow_paid": false,
    "respect_robots": true,
    "allow_login": false,
    "allow_paywalled": false
  }
}
```

The implementation may accept a compact form, but internal normalization must
produce these fields or explicit defaults.

## Search Provider Design

### Contract

v1.1 must define `BaseSearchProvider` as the canonical search provider contract.
If the current code still exposes `BaseProvider`, it should either be migrated
or kept as a compatibility alias that delegates to `BaseSearchProvider`.

Required provider operations:

- `provider_id()`
- `readiness(config) -> ProviderReadiness`
- `search(request) -> SearchResponse`
- `supported_constraints()`

Required `ProviderReadiness` fields:

- `provider`
- `available`
- `reason`
- `credential_status`
- `health_check_status`
- `quota_status`
- `constraint_support`
- `failure_code`
- `checked_at`
- `evidence`

Required failure codes:

- `PROVIDER_CREDENTIAL_MISSING`
- `PROVIDER_HEALTH_FAILED`
- `PROVIDER_QUOTA_EXHAUSTED`
- `PROVIDER_CONSTRAINT_UNSUPPORTED`
- `PROVIDER_RETIRED`

### Built-In Provider Targets

v1.1 should support the default broad provider plus an open-license provider
before release:

- Default broad web image provider: `serpapi_google_images`, backed by SerpApi
  Google Images with `SERPAPI_API_KEY` by default.
- An open-license provider: `commons`.

Additional providers can be added behind config without changing the
orchestrator.

### Scheduling

Multiple available providers are selected by weighted random scheduling.
Scheduling must preserve provenance:

- query id
- provider id
- provider rank
- search round
- raw result id
- normalized candidate id
- source page URL
- image URL

## Candidate Normalization and Quality

Candidate normalization must produce a canonical `CandidateRecord`:

- `candidate_id`
- `query_plan_id`
- `provider_id`
- `search_round`
- `rank`
- `image_url`
- `source_page_url`
- `thumbnail_url`
- `title`
- `snippet`
- `width`
- `height`
- `mime_type`
- `license_hint`
- `origin_candidate_ids`
- `dedupe_key`
- `provenance`

Deduplication keys should include normalized image URL, normalized source page
URL, provider-specific image identity, and artifact hash when available.

Candidate quality has two stages:

1. Mechanical validation:
   - Blocking metrics reject impossible, malformed, unsafe, or clearly
     irrelevant candidates.
   - Reference metrics are retained for subjective evaluation.
2. Subjective evaluation:
   - Must call production Qwen 3.5 VLM in production mode.
   - Must return an auditable decision.
   - Unavailable Qwen 3.5 VLM blocks production execution.

Only candidates accepted by both stages may enter retrieval batch planning.

## Retrieval Channel Design

v1.1 must treat retrieval as an artifact-producing subsystem. A URL fetch that
only returns bytes is not enough.

### Contract

`BaseRetrievalChannel` must return a `RetrievalArtifactResult` rather than only
a local image path.

Required operations:

- `channel_id()`
- `tier()`
- `readiness(config) -> RetrievalChannelReadiness`
- `retrieve_batch(batch) -> RetrievalBatchResult`

Required channel tiers:

| Tier | Purpose | Default |
|---|---|---|
| `normal_web_fetch` | Direct image fetch and source-page resolve | Enabled when safe |
| `self_hosted_service` | Open-source/self-hosted retrieval service | Disabled until configured |
| `paid_online_service` | Paid external retrieval service | Disabled unless explicit |

`normal_web_fetch` may include both direct HTTP image fetch and source-page
resolve fallback. The tier is still level 1 because it is the free local web
path.

### Retrieval Job

Each selected candidate becomes one or more retrieval jobs:

```json
{
  "retrieval_job_id": "ret-job-001",
  "candidate_id": "cand-001",
  "query_plan_id": "query-001",
  "target": {
    "target_type": "image",
    "primary_url": "https://example.com/image.jpg",
    "alternate_source_page_url": "https://example.com/page.html",
    "media_type": "image/*"
  },
  "requested_outputs": [
    "source_artifact",
    "source_sidecar",
    "content_summary",
    "task_report",
    "visual_description"
  ]
}
```

Retrieval batch size per attempt is `required_image_count * 2`. Short batches
are allowed only when there are not enough qualified candidates, and the
shortage must be recorded as a coverage gap.

### Required Artifact Result

An artifact-backed retrieval result must include:

- `retrieval_job_id`
- `candidate_id`
- `query_plan_id`
- `channel_id`
- `channel_tier`
- `retrieval_status`
- `local_artifact_path`
- `source_artifact_path`
- `source_sidecar_path`
- `content_summary_path`
- `task_report_path`
- `visual_description_path`
- `checksum_sha256`
- `content_type`
- `file_size_bytes`
- `image_dimensions`
- `media_type_match`
- `fetch_trace`
- `failure_reason`

`local_artifact_path` may be the same file as `source_artifact_path`, but both
fields must be explicit so package validation can distinguish missing evidence
from missing image bytes.

### Fallback

Fallback order:

1. Normal web fetch:
   - direct image URL
   - source page resolve
2. Self-hosted service
3. Paid online service

Fallback can advance only when policy permits it. Paid fallback requires an
explicit enablement decision and budget boundary.

Every failed attempt must record:

- channel id
- candidate id
- URL attempted
- failure category
- retryability
- policy reason
- timestamp

## Image Acceptance

A retrieved image counts as accepted only when both mechanical acceptance and
Qwen 3.5 VLM subjective evaluation pass.

Mechanical acceptance must include blocking checks for:

- artifact paths exist
- checksum present
- content type is image-compatible
- media type matches requested target
- file is non-empty and within configured size bounds
- dimensions meet quality policy
- no known access restriction or prohibited source violation
- job ownership matches candidate and query plan
- no metadata-only substitution

Reference metrics should include:

- dimensions
- file size
- source authority signals
- license/authorization hints
- source diversity
- visual description confidence
- negative-scope risk

Subjective image evaluation must call Qwen 3.5 VLM with the local artifact and
reference metrics. Production unavailability blocks the run.

## Delivery Qualification

A candidate may enter `delivered_images` only when all of the following hold:

- `fetch_status = "fetched"`
- `local_artifact_exists = true`
- `retrieval_status = "complete"`
- `local_artifact_path` is present
- `source_artifact_path` is present
- `source_sidecar_path` is present
- `content_summary_path` is present
- `task_report_path` is present
- `visual_description_path` is present
- `checksum_sha256` is present
- `media_type_match = true`
- mechanical image acceptance passed
- Qwen 3.5 VLM image evaluation passed
- candidate ownership is preserved
- no blocking negative-scope contradiction exists

Metadata-only, thumbnail-only, source-page-only, or summary-only results are
never delivered images.

## Orchestrator and Retry Loop

The v1.1 orchestrator must run the whole workflow per attempt:

```text
for full_attempt_count in 1..=4:
  recall candidates
  evaluate candidates
  retrieve selected batch
  evaluate images
  update accepted set
  if accepted_count >= required_image_count:
    return full_delivery
  if full_attempt_count < 4:
    prepare next retry with previous failures and accepted images excluded

return limited_delivery if accepted_count > 0 else blocked
```

Rules:

- `retry_count = full_attempt_count - 1`.
- Accepted images are carried across attempts.
- Previously failed candidates should not be retried through the same failed
  channel unless a different fallback path or changed policy exists.
- Search may be expanded on retry by provider, query text, round, or rank
  window.
- Limited delivery must list exactly why each gap remains.

## Delivery Package v1.1

The package root must contain:

```text
package/
  image-recalls.json
  retrieved-images.json
  coverage-report.json
  retrieval-manifest.json
  package-summary.json
  delivery-report.json
  validation.json
  review.json
  handoff-report.json
  images/
  evidence/
  diagnostics/
```

Required status values:

- `passed`: requested count reached and all gates pass.
- `partial`: useful images exist but one or more gates or requested counts are
  incomplete after retries.
- `blocked`: no traceable artifact-backed image delivery is possible.

`retrieval-manifest.json` must track each candidate status:

```text
recalled -> selected_for_fetch -> fetched -> delivered
                                  -> rejected
                                  -> failed
```

The manifest must preserve links among query plan, provider, candidate,
retrieval job, local artifact, evidence files, acceptance decision, and delivery
status.

## Validation and Review

v1.1 must include deterministic package validation before release acceptance.

Minimum validation checks:

- Required package files exist.
- JSON files conform to schema or typed serialization contract.
- Every delivered image has a valid retrieval job id.
- Every delivered image has all required artifact paths.
- Every required artifact path exists on disk.
- Every delivered image has checksum and media-type evidence.
- No delivered image is metadata-only.
- Coverage counts match delivered images and gaps.
- Retry counters are consistent.
- Provider readiness evidence exists.
- Qwen 3.5 VLM candidate and image decisions exist for delivered images.
- Sensitive credentials are absent from package files.

Semantic review remains separate from deterministic validation and should
produce `review.json` with pass/revise/fail style verdicts.

## CLI Surface

Required v1.1 commands:

```bash
image-retrieval run --query-plan query-plan.json --output-dir out/
image-retrieval self-check --config config.toml
image-retrieval validate-package --package-dir out/package
```

Recommended commands:

```bash
image-retrieval providers --config config.toml
image-retrieval inspect-package --package-dir out/package
```

`run` must execute the full workflow. It must not stop at task planning.

## Configuration

v1.1 configuration must externalize:

- provider enablement
- provider weights
- provider credentials by environment variable name
- channel enablement
- channel fallback policy
- paid channel budget/confirmation
- robots/site-rule behavior
- Qwen 3.5 VLM endpoint/model configuration
- quality thresholds
- output path policy

Credentials must never be serialized into delivery packages, diagnostics, or
logs.

## Test Requirements

Required test scenarios:

- QueryPlan defaults and invalid input rejection.
- Candidate target is roughly `20 * required_image_count`.
- Retrieval batch target is `2 * required_image_count`.
- Provider missing credential blocks readiness.
- Weighted provider scheduler preserves provenance.
- Candidate dedupe keeps origin candidate ids.
- Metadata-only retrieval cannot be delivered.
- Missing source sidecar cannot be delivered.
- Missing summary cannot be delivered.
- Missing task report cannot be delivered.
- Direct image fetch success creates all required artifact fields.
- Source-page fallback success records both attempts.
- Paid channel disabled cannot be used.
- Qwen 3.5 VLM unavailable blocks production evaluation.
- Retry loop stops after one initial attempt and three retries.
- Limited delivery after retry exhaustion reports exact gaps.
- Package validation fails when any delivered image artifact is missing.
- Sensitive credential redaction is enforced.

Expected Rust verification commands:

```bash
cargo fmt --all -- --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test --all
```

## Release Gates for v1.1

These decisions must be closed before declaring v1.1 complete:

1. Implement and verify the default SerpApi Google Images search provider.
2. Select the built-in provider list and restricted/legacy provider policy.
3. Implement and verify the `qwen_3_5_vlm` adapter with `QWEN_API_TOKEN`,
   externalized endpoint/model config, response validation, and real-service
   smoke evidence.
4. Decide paid retrieval channel enablement and budget boundary.
5. Decide robots.txt / site-rule compliance behavior.
6. Define authorization blocking rules.
7. Calibrate or explicitly waive quality thresholds.

If any gate remains unresolved, v1.1 may still ship as an experimental build,
but it must not be described as having completed the expected product target.

## Current Implementation Delta

The existing codebase already contains useful foundations:

- Rust CLI project structure.
- QueryPlan and task planning domain models.
- Search and retrieval traits/registries.
- Fixture search and retrieval paths.
- Candidate and image quality gate boundaries.
- Delivery package builder.
- Self-check module.
- Fixture E2E tests.

The main v1.1 gaps are:

- `run` must execute the full workflow, not only planning.
- The default SerpApi Google Images provider must be implemented behind
  configuration.
- Retrieval must produce auditable artifacts, not only downloaded bytes.
- Retrieval fallback channels must be implemented and policy-gated.
- Qwen 3.5 VLM production adapter, endpoint/model configuration, and smoke evidence must be implemented.
- Delivery package must use the v1.1 canonical evidence contract.
- Validation must reject missing artifact evidence and metadata-only delivery.
- Real service verification must prove at least one artifact-backed accepted
  image.
